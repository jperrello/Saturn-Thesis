    \vspace{0.25in}

    \authorname

\vspace{0.5in}

Generative AI services require users to obtain API keys, manage credentials, and configure endpoint URLs for each application and provider. These per-user, per-application configuration steps create barriers to access that disproportionately affect resource-constrained populations. This thesis presents Saturn, a protocol for zero-configuration AI service discovery on local networks using multicast DNS (mDNS) and DNS-based Service Discovery (DNS-SD). 

Saturn servers advertise OpenAI-compatible AI endpoints with ephemeral, time-limited credentials on the local network. Clients discover these services automatically without manual configuration. We advance three claims: (1) zero-configuration discovery is feasible via mDNS/DNS-SD, (2) network-level provisioning reduces configuration effort compared to manual setup, and (3) information exposure is documented and bounded.

We demonstrate Saturn across six implementations spanning four programming languages and show practical deployment on consumer hardware. The protocol extends zero-configuration networking---proven for printers and speakers---to generative AI services, enabling institutions to provision shared AI access at the network level and reducing total configuration burden from scaling with the number of users to scaling only with the number of administrators.
}

\chapter{Introduction}
% ---------------------------------------------------------\section{The Problem: AI Access Inequity}

In 1954, Lewis Strauss predicted that atomic energy would become ``too cheap to meter''~\cite{strauss1954}. Seventy years later, generative AI carries a similar promise---and a similar gap between rhetoric and reality. ChatGPT Plus costs \$20 per month~\cite{chatgpt-plus}. Anthropic's Claude Pro costs the same~\cite{claude-pricing}. Users who need multiple providers or higher usage limits pay \$100 or more per month. These are not edge cases; they represent the standard path to accessing the most capable AI systems available today.

The cost barrier alone would warrant attention, but access to generative AI demands more than a subscription. Each application that consumes AI must be configured independently: an API key obtained from a provider, an endpoint URL, and often a specific model identifier. Syed et al.~\cite{syed2025aiaas} document this as the standard architecture of AI-as-a-Service, where platforms expose REST APIs behind authentication tokens that individual users must obtain, manage, and rotate. A developer using three AI-powered tools---a chat interface, a coding assistant, and a transcription service---must configure credentials for each one separately.

This per-user key management model is not merely inconvenient; it is broken. Meli et al.~\cite{meli2019secrets} found that over 100,000 public GitHub repositories contained leaked API secrets, with 81\% of those secrets never revoked even after notification. The best automated scanning tools they evaluated detected only 25\% of leaked credentials. Static API keys, the dominant access mechanism for AI services, create a class of security vulnerabilities that current tooling cannot adequately address.

The consequences fall unevenly. Bassignana et al.~\cite{bassignana2025aigap} demonstrate that socioeconomic status correlates with the quality of interactions people have with language technology---those with fewer resources extract less value even when they gain access. Gabriel~\cite{gabriel2024educational} identifies generative AI as a widening force in educational inequality, where students who can afford subscriptions gain compounding advantages in writing, research, and problem-solving. Capraro et al.~\cite{capraro2024impact} find that generative AI's economic benefits accrue disproportionately to those who already hold advantages. Costa et al.~\cite{costa2024democratization} frame accessibility and usability as foundational dimensions of AI democratization, arguing that technical barriers reinforce existing power asymmetries. The current model---individual subscriptions, per-app configuration, manual credential management---excludes precisely the populations most likely to benefit from shared access.

\section{The Vision: AI as Network Infrastructure}

When Apple introduced Bonjour in 2002, a user who connected to a WiFi network could print within seconds---no driver downloads, no IP addresses, no configuration dialogs. The underlying protocol, Multicast DNS (mDNS) combined with DNS-based Service Discovery (DNS-SD), allowed printers to announce their presence on the local network and clients to find them automatically. Guttman~\cite{guttman2001zeroconf} described the goal of zero-configuration networking as enabling ``direct communications between two or more computing devices via IP'' with no manual setup, and RFC~1122~\cite{rfc1122} articulated the principle even earlier: ``It would be ideal if a host implementation of the Internet protocol suite could be entirely self-configuring.''

Generative AI services have no equivalent. Every AI application today requires users to manually locate a provider, create an account, obtain credentials, and configure each tool individually. Yet the technical requirements for discovery are similar to those that printers solved twenty years ago: a service exists on a network, it has capabilities that clients need to know about, and the connection requires minimal metadata.

Institutional networks already provision shared resources this way. University eduroam networks grant students WiFi access across hundreds of campuses without per-institution accounts. Campus print services discover printers automatically. The conceptual step from ``join the network, get the printer'' to ``join the network, get the AI'' is small. The implementation gap is what Saturn addresses.

\section{Thesis Statement}

We can and should provision generative AI at the network level. Saturn demonstrates this is technically achievable using existing zero-configuration protocols, enabling shared access models where joining a network means gaining AI capabilities. This thesis advances three claims. First, zero-configuration network protocols (mDNS and DNS-SD) can provision AI services without end-user configuration. Second, network-level provisioning reduces total configuration effort by centralizing setup in a single administrator rather than distributing it across every user. Third, the security and privacy trade-offs of broadcast-based AI discovery are documented in existing literature and can be mitigated without destroying the zero-configuration property.

\section{Saturn and This Thesis}

Saturn is a protocol and family of implementations for zero-configuration AI service discovery. Servers advertise OpenAI-compatible AI endpoints on the local network using mDNS/DNS-SD under the service type \texttt{\_saturn.\_tcp.local.}; clients discover these services automatically and connect without configuration. The protocol specifies a beacon architecture: servers broadcast metadata about where AI services live, rather than proxying traffic through a central point. This eliminates single points of failure and preserves the direct relationship between client and AI provider.

We implemented Saturn six times across four programming languages. The core Python package provides both server and client libraries. A TypeScript provider integrates with Vercel's AI SDK framework. An Open WebUI plugin adds discovery to an existing chat interface. A Model Context Protocol server exposes Saturn to AI coding assistants. A Lua-to-Python bridge brings discovery into VLC media player. A Rust implementation runs on a GL.iNet travel router alongside the device's own DHCP and DNS services---the same class of hardware that provisions IP addresses now also provisions AI.

The chapters that follow establish the service discovery landscape and AI access patterns that motivate Saturn's design, then present the protocol specification, its six implementations, and deployment scenarios including a design fiction and a 42-application integration analysis. Saturn demonstrates that the infrastructure for shared AI access does not need to be invented---it needs to be assembled from protocols that have worked for decades.

This thesis was developed with substantial support from Claude (Anthropic), a large language model serving two roles: authoring the moons knowledge graph through iterative prompting and providing iterative feedback on coherence and writing. We document this openly because Saturn argues generative AI should be accessible to everyone---and that commitment demands transparency about how we conducted our research. Concealing AI's role would contradict the equity argument we advance. A full account appears in Appendix B.

\chapter{Background}
% ------------------------------------------------------------

Devices on a local network need to find each other. A laptop locates a printer; a phone finds a speaker; a coding tool connects to a debugger. This problem---service discovery---has generated competing protocols over three decades, each reflecting different assumptions about platform support, scope, and configuration overhead. This chapter examines those protocols, explains why mDNS and DNS-SD fit Saturn's requirements, describes the network layer where Saturn operates, and surveys the AI service landscape Saturn targets.

\section{Service Discovery on Local Networks}

When two devices share a network segment, neither knows the other exists. Service discovery protocols solve this bootstrapping problem by defining how devices announce their presence and how clients search for available services.

\subsection{NetBIOS}

Network Basic Input/Output System (NetBIOS) provided name resolution on local networks before DNS became ubiquitous~\cite{rfc1001}. Windows machines used NetBIOS over TCP/IP to discover file shares and printers via broadcast messages. NetBIOS remains present in modern Windows networks but has no native support on macOS or Linux. Microsoft began replacing it with mDNS in Windows~10, signaling its trajectory toward deprecation. For Saturn, NetBIOS fails on two counts: it cannot operate cross-platform, and it provides name resolution only---no mechanism for service metadata such as model lists, API types, or feature capabilities.

\subsection{Universal Plug and Play}

Universal Plug and Play (UPnP) combines the Simple Service Discovery Protocol (SSDP) for multicast device discovery with a SOAP/XML framework for device control~\cite{upnp2015}. Routers use UPnP for automatic port forwarding; media servers use it to stream content. UPnP extends far beyond discovery into device interaction, introducing complexity Saturn does not need, and carries a documented history of security vulnerabilities including remote code execution and unauthorized port forwarding. Apple, Google, and Microsoft have shifted to mDNS/DNS-SD for new service discovery features, reflecting UPnP's declining role.

\subsection{DLNA}

The Digital Living Network Alliance (DLNA) built a media streaming standard on top of UPnP, enabling televisions to discover and play content from network media servers~\cite{dlna2017}. DLNA inherits UPnP's complexity and restricts its scope to audio and video. The DLNA consortium dissolved in 2017. Saturn requires general-purpose service metadata exchange, not media transport.

\subsection{WS-Discovery}

Web Services Dynamic Discovery (WS-Discovery) uses multicast SOAP/XML messages for service advertisement, primarily in enterprise environments~\cite{wsdiscovery}. Windows uses WS-Discovery alongside mDNS for printer discovery. The protocol's SOAP/XML encoding imposes substantial overhead for lightweight metadata exchange, and its library ecosystem is smaller and less portable than mDNS implementations. WS-Discovery targets managed networks with IT infrastructure---not the consumer and educational environments Saturn serves.

\subsection{Why mDNS/DNS-SD}

mDNS and DNS-SD satisfy Saturn's requirements where the alternatives do not. mDNS resolves names across platforms without infrastructure---no server, no accounts, no configuration. DNS-SD advertises services using standard DNS record types, carrying the metadata Saturn needs in a format every major operating system already supports. The same protocol pair powers AirPrint, AirPlay, Chromecast, and Spotify Connect. Guttman~\cite{guttman2001zeroconf} defined zero-configuration networking as enabling ``direct communications between two or more computing devices via IP'' without manual setup; mDNS/DNS-SD is the dominant realization of that vision. The next section (2.2) explains both protocols in detail.

\subsection{DHCP}

The Dynamic Host Configuration Protocol (DHCP) is Saturn's closest analogy at the network layer~\cite{rfc2131}. A device joining a network broadcasts a DISCOVER message; a DHCP server responds with an OFFER containing an IP address, gateway, DNS server, and lease duration; the client acknowledges, and connectivity begins without manual configuration.

Saturn and DHCP share a philosophy---join the network, receive the service---but operate at different layers. DHCP runs once, at the moment a device joins the network. It assigns a single IP address along with a gateway, DNS server, and lease duration, then its work is done. The protocol operates at the system level, binding network interfaces with root or administrative privileges, and carries only enough metadata to bring a device online. Saturn's requirements differ in timing, multiplicity, privilege, and payload. AI services appear and disappear dynamically, so discovery must be continuous rather than one-shot. A network may host dozens of services with different models and capabilities, so discovery must enumerate many results rather than a single assignment. Saturn runs in userspace with no elevated privileges and carries rich metadata---model lists, API types, priorities, ephemeral credentials---through DNS records rather than DHCP options.

The two are complementary rather than competing. DHCP configures the network. Saturn configures AI services on that network. On Saturn's GL.iNet router deployment, both run on the same hardware---the device assigns IP addresses via DHCP while simultaneously advertising AI endpoints via Saturn.

\section{mDNS and DNS-SD}
\label{sec:mdns-dnssd}

\subsection{Multicast DNS}
\label{sec:mdns-detail}

Multicast DNS (mDNS), specified in RFC~6762~\cite{rfc6762}, resolves names on local networks without a DNS server. Rather than querying a designated server, mDNS clients multicast queries to all devices on the link. Any device that owns the queried name responds directly.

Queries travel to the multicast address 224.0.0.251 on port 5353 (ff02::fb for IPv6). Names use the \texttt{.local} top-level domain---\texttt{my-server.local}, for instance. When a client resolves \texttt{my-server.local}, it multicasts the query, and the owning device replies with its IP address. No central authority mediates.

mDNS inherits the DNS record type system: A records (IPv4 addresses), AAAA records (IPv6), SRV records (service locations), TXT records (key-value metadata), and PTR records (service enumeration). The wire format is identical to conventional DNS; only the transport differs with multicast instead of unicast, and link-local scope instead of global.

Siddiqui et al.~\cite{siddiqui2012zeroconf} identify mDNS as one of three pillars of zero-configuration networking, alongside link-local addressing~\cite{rfc3927} and DNS-SD. Together, these protocols allow devices to obtain addresses, resolve names, and discover services without pre-existing infrastructure.

\subsection{DNS-Based Service Discovery}
\label{sec:dnssd-detail}

DNS-based Service Discovery (DNS-SD), specified in RFC~6763~\cite{rfc6763}, defines how clients enumerate services by type. While mDNS resolves names to addresses, DNS-SD finds all instances of a given service on the network.

DNS-SD chains three record types. A PTR record maps a service type---such as \texttt{\_saturn.\_tcp.local.}---to the names of all instances offering that service. An SRV record for each instance provides hostname and port. A TXT record carries key-value metadata about capabilities. Discovery follows the sequence: query PTR for the service type, receive instance names, then query SRV and TXT for each instance.

DNS-SD can run over mDNS for link-local discovery or over conventional unicast DNS for wide-area use. Saturn uses mDNS transport exclusively, confining discovery to the local network segment.

\subsection{TXT Records and Metadata}

TXT records carry key-value pairs as DNS record data. Each string has a maximum length of 255~bytes, imposed by the DNS wire format. A single TXT record can contain multiple strings, but each must fit within this limit independently.

Saturn uses TXT records to advertise: protocol version, deployment name, API type, base URL, priority level, ephemeral key, rotation interval, and supported features. Clients select among discovered services using this metadata alone, without additional network requests beyond the DNS-SD queries. The 255-byte limit constrains individual field sizes---ephemeral keys must fit within approximately 240 usable characters---but accommodates all metadata Saturn requires.

\subsection{Implementations: Bonjour and Avahi}

Apple released Bonjour in 2002 as the reference mDNS/DNS-SD implementation, bundled with macOS and available for Windows~\cite{bonjour}. Bonjour powers AirPrint, AirPlay, and AirDrop. On Linux, the Avahi daemon provides equivalent functionality and ships as a default service on most distributions~\cite{avahi}. Microsoft added native mDNS responder support in Windows~10.

Every major desktop and mobile operating system can therefore participate in mDNS/DNS-SD discovery without additional software. This cross-platform foundation means any application using system mDNS APIs---regardless of programming language or framework---can discover Saturn services.

\subsection{Adoption in New Domains}

mDNS/DNS-SD originated with printer and file share discovery but has expanded into domains its designers did not anticipate. AirPlay streams audio and video to speakers and televisions. Chromecast casts media from phones and laptops. Spotify Connect discovers playback devices across the local network.

Siljanovski et al.~\cite{siljanovski2014mdns} demonstrated that mDNS serves Internet of Things environments, proposing it as a discovery layer for heterogeneous sensor networks. VS~Code uses mDNS to discover remote debugging targets, enabling developers to attach debuggers to other machines without entering IP addresses manually~\cite{vscode-remote}. A community contributor proposed adding mDNS service advertisement to Ollama, a widely deployed local AI inference server, in an unmerged pull request~\cite{ollama-mdns-pr}---one of the earliest attempts to apply zero-configuration discovery to AI services, and a direct precedent for Saturn's approach.

Saturn continues this trajectory. Where AirPrint discovers printers and AirPlay discovers speakers, Saturn discovers AI endpoints. The service type changes---\texttt{\_saturn.\_tcp} instead of \texttt{\_ipp.\_tcp}---but the mechanism is identical.

\subsection{Privacy Considerations}

mDNS operates by multicast: every query and response reaches all devices on the link. Kaiser and Waldvogel~\cite{kaiser2014privacy} demonstrated that passive observers can enumerate all services on a network, track device presence over time, and correlate service metadata with users. K\"{o}nings et al.~\cite{konings2013devicenames} measured device names in the wild and found that 59\% contained real user names, making mDNS announcements a source of personally identifiable information.

These findings apply directly to Saturn. A passive observer on a Saturn-enabled network sees which AI services exist, what models they offer, and potentially who advertises them. Kaiser and Waldvogel~\cite{kaiser2014efficient} proposed privacy-preserving extensions to DNS-SD using hash-based service instance names, which reduce what observers learn without breaking discovery. Chapter~7 analyzes Saturn's specific threat model in the context of these known vulnerabilities.

\section{The AI Service Landscape}

Saturn targets applications that consume AI text completion and transcription services. These applications share a configuration pattern: each requires users to specify an API endpoint, authenticate with credentials, and select a model.

\subsection{Hosted Chat Applications}

Self-hosted chat interfaces provide a chat like experience backed by user-controlled AI endpoints. Open WebUI~\cite{open-webui} connects to Ollama~\cite{ollama} instances or any OpenAI-compatible API through a web interface. LibreChat~\cite{librechat} supports multiple providers simultaneously, letting users switch between OpenAI, Anthropic, and local models within a single session. Jan~\cite{jan-ai} runs models locally on the desktop while also supporting remote API connections.

Each application requires the user to configure at least one backend: an API base URL, an authentication key, and a model identifier. Users who run multiple chat interfaces multiply this effort. The configuration is static: if a backend changes its address, port, or credentials, every connected application must be updated manually.

\subsection{Coding Agents and Assistants}

AI-powered development tools increasingly support custom endpoints rather than locking users to a single provider. Cursor~\cite{cursor} routes completions through any OpenAI-compatible endpoint, enabling developers to use local models or alternative providers. Continue~\cite{continue}, an open-source coding assistant, connects to arbitrary OpenAI-compatible APIs. Both require users to configure endpoint URLs and API keys per provider.

These tools illustrate the same pattern at a different layer: developers must configure AI access individually for each tool in their workflow, even when all tools ultimately query the same backend.

\subsection{Voice Typing and Transcription}

Voice-to-text tools demonstrate a distinct AI consumption pattern where two models collaborate in a pipeline. Most use an on-device speech recognition model like OpenAI's Whisper~\cite{whisper} to transcribe audio locally. A language model then cleans the raw transcript: correcting grammar, removing filler words, and formatting text.

SuperWhisper~\cite{superwhisper} runs Whisper entirely on-device, ensuring no audio leaves the user's machine at the cost of accepting whatever quality the local model achieves. Wispr Flow~\cite{wisprflow} adds a cloud-based language model to clean and reformat transcripts, delivering polished output for a monthly subscription. MacWhisper~\cite{macwhisper} offers local transcription with a one-time purchase, avoiding recurring costs but limiting users to bundled models.

This pipeline captures the fragmentation Saturn addresses. A user wanting voice transcription with AI cleanup chooses between a subscription for cloud processing, a standalone tool with fixed capabilities, or manually configuring both a local Whisper instance and a separate language model. Each option encodes infrastructure decisions---which model, where it runs, how to authenticate---into the individual user's configuration.

\chapter{Design}
% ------------------------------------------------------------

Chapter~2 established that mDNS and DNS-SD provide cross-platform, infrastructure-free service discovery, and that every AI application today requires manual endpoint configuration. This chapter specifies how Saturn bridges that gap: the properties it must achieve, the roles it defines, its core abstractions, the protocol in full detail, and the architectural decisions that shaped the design.

\section{Goals}

Saturn pursues four design goals, each responding to a problem identified in Chapters~1 and~2.

End users should gain AI access by simply joining a network---no software installation, no account creation, no API key entry, no model selection. The configuration effort required of consumers should be zero if Saturn achieves its primary goal.

The protocol makes no assumptions about what provides AI inference. Any service that exposes an OpenAI-compatible REST API participates equally, whether it runs on a cloud provider, a local GPU, or a travel router. Saturn binds to an interface standard, not a provider or deployment topology.

Rather than distributing configuration burden across every user and every application, one administrator configures services for all users on a network. A university IT department, a hobbyist running a local model at home, or a household member sharing cloud API access with family---each configures once, and every device on the network benefits. This shifts total configuration effort from scaling with the number of users and applications to scaling only with the number of services.

When AI services require API keys, those keys must resist theft. A participant who joins a network, captures a broadcast credential, and leaves should find that credential expired by the time they attempt to use it elsewhere. Saturn achieves this through forced rotation: cloud-facing services rotate credentials on a fixed schedule and announce both the key and its rotation interval, constraining the window an attacker has to exploit a captured credential.

\section{Audiences}

Four roles interact with Saturn. Each encounters a different surface of the protocol.

LLM inference providers operate the AI services that Saturn discovers. Providers such as OpenAI, Anthropic, and OpenRouter already expose OpenAI-compatible REST APIs; Saturn requires no changes to their endpoints. Their incentive is distribution: a provider whose API is discoverable via Saturn reaches every user on every Saturn-enabled network without negotiating individual accounts or institutional contracts.

IT administrators perform all Saturn configuration. They obtain API keys from providers, deploy beacon processes, name services, and assign priorities. Every property that makes Saturn zero-configuration for end users---automatic discovery, credential distribution, service selection---emerges from an administrator's deliberate setup. The administrator is Saturn's primary user.

End users consume AI through applications they already use. They do not know what an API key is, what model serves their request, or that Saturn exists. Their experience mirrors AirPlay: join a WiFi network, open a compatible application, find AI services available immediately---exactly as they find printers or speakers. When Saturn works, it is invisible.

Application developers integrate Saturn discovery into their software. Their motivation is economic: they want to offer AI-powered features without subsidizing API costs for every user. By adding Saturn support, a developer delegates credential management and provider selection to the network, writing discovery code once while any Saturn-enabled network supplies the backend.

\section{Concepts}

This section defines four abstractions that the protocol specification (Section~3.4) depends on.

\subsection{Endpoints}

A Saturn endpoint is any HTTP service that implements the OpenAI-compatible REST API---the de facto standard interface for AI text generation, originally defined by OpenAI and now implemented by Ollama, OpenRouter, LiteLLM, vLLM, and dozens of other providers. The API specifies a JSON request/response format with a messages array, model identifier, and generation parameters.

Saturn adopts this existing interface rather than defining its own because most AI-consuming applications already support it. An application with a ``custom OpenAI endpoint'' setting works with Saturn without code changes. The protocol does not prescribe who runs the endpoint or where: a cloud provider, a university data center, or a single-board computer running quantized models all qualify, provided they implement the expected HTTP paths (Section~3.4.3).

\subsection{Beacons}

A beacon is a process that announces an AI endpoint on the local network via mDNS/DNS-SD. It registers a service under \texttt{\_saturn.\_tcp.local.}, publishes metadata in DNS TXT records, and responds to discovery queries. A beacon does not proxy API traffic---clients connect directly to the advertised endpoint. The beacon tells them where to connect and how to authenticate; it never handles their requests.

This separation matters for three reasons. First, a beacon cannot observe user prompts or model responses, because API traffic never flows through it. Second, a beacon failure does not sever existing connections; clients that have already discovered an endpoint continue communicating with it directly. Third, beacons require minimal resources---DNS registration and credential management, not request forwarding---which enables deployment on constrained hardware such as travel routers.

Saturn defines two beacon types based on deployment context. A \emph{cloud beacon} announces a remote AI provider and distributes ephemeral API keys that grant temporary access to that provider's API. A \emph{local beacon} announces a service running on the local network and monitors its health, deregistering the service if it becomes unreachable.

\subsection{Priorities}

Each Saturn service advertises a numeric priority in its TXT record. Lower values indicate higher preference: a service at priority~1 is selected before a service at priority~50. Clients sort discovered services by priority and connect to the lowest available value.

Priority is the administrator's policy mechanism. An IT administrator who wants users to prefer local inference sets the local beacon to priority~1 and the cloud beacon to priority~50. Users receive local inference by default and fall back to the cloud only when the local service is unavailable. Saturn uses explicit numeric priority rather than round-robin or random selection because the administrator---not the client---controls routing policy. The semantics mirror DNS SRV record priority: lower values mean higher preference, and clients process services in ascending order.

\subsection{Ephemeral Keys}

Cloud AI providers authenticate requests with API keys. When Saturn distributes access to a cloud provider, the beacon broadcasts a temporary key rather than the administrator's permanent credentials. This key has a fixed lifetime---typically 10~minutes---and the beacon generates a replacement before the current key expires, typically every 5~minutes. The overlap window ensures zero-downtime transitions: clients using the old key continue working while new clients receive the replacement. After a brief grace period, the beacon deletes the expired key from the provider's system.

This rotation schedule serves the credential transience goal. A key captured from a network broadcast expires within minutes. An attacker who records the key and leaves the network finds it revoked before they can use it elsewhere. The approach parallels Kerberos ticket distribution: time-limited credentials issued for a specific session scope, applied here at the application layer through provider key provisioning APIs.

Ephemeral keys appear only in cloud beacon deployments. Local services that require no authentication---an Ollama instance on a private network, for example---advertise no key. The TXT record schema (Section~3.4.2) distinguishes these cases through the \texttt{features} field. This mechanism requires that the AI provider support programmatic key creation and deletion; providers without provisioning APIs cannot participate in ephemeral key distribution.

\section{Protocol Specification}

This section specifies the Saturn protocol in sufficient detail for independent implementation. Two implementors reading only this section should produce interoperable systems.

\subsection{Service Type}

Saturn services register under the DNS-SD service type \texttt{\_saturn.\_tcp.local.}. The underscore-prefixed format follows RFC~6763 naming conventions for DNS-SD service types~\cite{rfc6763}. The \texttt{\_tcp} designator indicates that clients connect via TCP after discovery. The \texttt{.local.} suffix confines discovery to the link-local multicast domain defined by mDNS~\cite{rfc6762}.

\subsection{Record Structure}

Saturn uses the DNS-SD record chain described in Section~\ref{sec:dnssd-detail}. Three DNS record types work together for each service instance:

A \textbf{PTR record} maps the service type \texttt{\_saturn.\_tcp.local.} to the names of all Saturn instances on the network. Clients query this record to enumerate available services.

An \textbf{SRV record} for each instance provides a hostname and port number. The hostname resolves to an IP address via mDNS.

A \textbf{TXT record} for each instance carries key-value metadata: deployment type, API compatibility, priority, and---for cloud beacons---ephemeral credentials. Clients select among discovered services using TXT metadata alone, without additional network requests beyond the initial DNS-SD queries.

Table~\ref{tab:txt-schema} specifies the TXT record fields that constitute the Saturn protocol.

\label{tab:txt-schema}
\begin{tabular}{l l p{3in}}
\hline
\textbf{Field} & \textbf{Presence} & \textbf{Description} \\
\hline
\texttt{version} & Required & Protocol version (currently \texttt{1.0}). \\

\subsection{Discovery Process}

Client discovery follows four steps:

\textbf{Step~1: Browse.} The client multicasts a DNS-SD PTR query for \texttt{\_saturn.\_tcp.local.}, requesting the names of all Saturn service instances on the local network.

\textbf{Step~2: Resolve.} For each discovered instance, the client queries the corresponding SRV and TXT records. The SRV record yields a hostname and port; the TXT record provides metadata including deployment type, API base URL, priority, and---for cloud beacons---an ephemeral credential.

\textbf{Step~3: Sort.} The client orders discovered services by the \texttt{priority} field in ascending order. Ties may be broken by implementation-defined criteria such as the \texttt{cost} field or random selection.

\textbf{Step~4: Select.} The client issues a \texttt{GET /v1/health} request to the highest-priority service. If the service responds, the client connects. If not, the client proceeds to the next service in priority order. Discovery fails only when no service responds.

When a connected service becomes unavailable---detected by a failed health check or a failed API request---the client re-executes the discovery process from Step~1.

\section{Architecture Decisions}

Four architectural decisions shape Saturn. Each represents a deliberate choice against a viable alternative.

Rather than routing traffic through an intermediary, Saturn announces where AI services live and lets clients connect directly. A proxy architecture would simplify client integration---clients need only one URL---but introduces a single point of failure, grants the operator visibility into every user prompt and response, and must handle the entire throughput of the backends it fronts. The beacon pattern requires clients to discover endpoints, but eliminates these liabilities. Discovery is automatable: clients execute it once at startup and repeat only on service failure, making the trade-off acceptable.

Saturn uses link-local multicast for discovery rather than a centralized service catalog. A registry server offers faster lookups and cross-subnet visibility but reintroduces infrastructure dependency: someone must operate, maintain, and secure it. mDNS requires no infrastructure beyond the network itself. This confines Saturn to a single broadcast domain---devices must share a multicast segment---but matches the deployment contexts Saturn targets: home networks, campus WiFi, and small offices where all participants share a network segment.

The protocol itself is decoupled from implementation. Saturn specifies a service type, a TXT record schema, and endpoint expectations, but does not mandate a programming language, an mDNS library, or a deployment platform. Independent implementations interoperate by construction: any process that registers under \texttt{\_saturn.\_tcp.local.} with conforming TXT records and serves the expected endpoints is a valid Saturn service.

Saturn targets text completion and chat endpoints exclusively. Image generation, audio synthesis, and embedding computation use different API interfaces and would require extending both the TXT schema and endpoint expectations. Text completion is chosen because it is the most widely deployed AI capability and the one the OpenAI-compatible API standardizes. Establishing zero-configuration discovery on this single modality demonstrates feasibility; extending to additional modalities requires no changes to the underlying mDNS/DNS-SD mechanism, only new service types and TXT field definitions.

\subsection{Server Architecture}

The core Saturn package provides shared libraries for beacon registration, mDNS query handling, and TXT record encoding/decoding. The server architecture centers on a beacon process that registers a service under \texttt{\_saturn.\_tcp.local.} using the system's native mDNS responder (Bonjour on macOS, Avahi on Linux) or direct UDP multicast for platforms without daemon support. The beacon maintains TXT records with current metadata, refreshes service registration periodically to prevent timeout, and handles credential rotation for cloud-facing beacons.

\subsection{Client Discovery}

Client code follows the four-step discovery process from Section~3.4.4: browse for \texttt{\_saturn.\_tcp.local.}, resolve SRV and TXT records, sort by priority, and probe each service with a health check. The client library exposes a simple API: call \texttt{discover()} to return available services with metadata, then \texttt{connect(service)} to establish a client authenticated with the provided ephemeral key. Discovered services cache for 60 seconds; clients refresh when they fail to connect to a previously discovered service.

\section{Server Types}

\subsection{Ollama Server}

Saturn's Python reference implementation includes a built-in beacon for Ollama, a widely deployed local inference engine. The Ollama beacon discovers running instances via port scanning on port 11434 or explicit configuration, verifies they respond to health checks, and advertises them via mDNS. Models enumerated from Ollama propagate directly into Saturn TXT records, keeping model lists synchronized automatically.

\subsection{OpenRouter Server}

The OpenRouter beacon provisions cloud-based inference by distributing ephemeral API keys from the administrator's OpenRouter account. OpenRouter supports programmatic key creation and deletion, enabling Saturn's credential rotation strategy. The beacon authenticates to OpenRouter's key provisioning API, creates 10-minute-lifetime keys on a schedule, and broadcasts each key in TXT records until replacement arrives.

\subsection{Fallback Server}

A fallback beacon serves requests directly, returning hardcoded models and demonstration responses. This server serves testing and documentation purposes, enabling developers to validate Saturn discovery without requiring Ollama or cloud credentials.

\subsection{Subprocess Bridge}

Saturn's design assumption---that applications natively support mDNS discovery---does not hold for most software. Most applications were built before Saturn existed. To prove Saturn works in production applications, we built a subprocess bridge pattern: a lightweight proxy that performs discovery on behalf of an application that cannot do it natively.

The VLC media player integration demonstrates this pattern. VLC is one of the world's most widely installed desktop applications, making it an ideal test case. End users do not have API keys, configuration files, or technical knowledge. If Saturn can bring AI discovery into VLC with zero user configuration, it proves the approach scales beyond developer tools.

\subsubsection{User Experience}

Installation requires only copying a directory to VLC's extensions folder and restarting. Activation is a single menu click: View $\to$ Extensions $\to$ Saturn Chat. No environment variables, no Python installation, no API key management.

When a user activates Saturn Chat, the extension automatically extracts metadata from the currently playing media---title, artist, album, genre, current timestamp---using VLC's native metadata API. A dialog window appears with two dropdowns: one for discovered services, one for available models. The user types a question about the media and receives a styled response. All discovery and model enumeration happen automatically in the background.

This interaction pattern mirrors traditional zero-config discovery: a user plugs in a USB printer and it appears in the system's printer list without driver installation or configuration. Here, a user opens VLC and AI services appear in a dropdown without credential entry or endpoint specification.

\subsubsection{Technical Architecture}

The integration uses a two-layer architecture to work around VLC Lua's limitations. VLC's extension system supports only Lua scripting with restricted HTTP capabilities (GET requests only, no POST). Lua also lacks native mDNS support.

Layer 1: A Python/FastAPI bridge service (`vlc\_discovery\_bridge.py`) runs as a background subprocess launched by the Lua extension. The bridge performs mDNS discovery using the same `dns-sd` subprocess calls as the core Saturn client, browses `\_saturn.\_tcp.local.`, resolves service addresses, maintains health checks, and exposes an OpenAI-compatible REST API on localhost (127.0.0.1). When the Lua extension activates, it launches the bridge with automatic port assignment. The bridge writes its port to a file that the extension reads, establishing inter-process communication without hardcoding ports.

Layer 2: The Lua extension (`saturn\_chat.lua`) creates a VLC dialog window, manages user interaction, formats requests as URL-encoded JSON payloads (working around Lua's GET-only limitation), and sends them to the localhost bridge via HTTP. When the user deactivates the extension, the bridge is terminated.

The bridge is packaged into a standalone executable using PyInstaller, eliminating the Python dependency for end users. Cross-platform backgrounding handles Windows (`start /B`), macOS, and Linux process launching. The entire installation is a directory copy; deployment requires no system-level configuration.

\subsubsection{Significance}

The VLC integration serves three purposes in Saturn's thesis argument. First, it proves that AI provisioning can happen at the network layer like print services---not merely in CLI tools or developer IDEs, but in applications designed and used by non-technical people. Second, the VLC bridge uses identical mDNS discovery code as all other Saturn components: `\_run\_dns\_sd\_discovery()` calls `dns-sd -B \_saturn.\_tcp local` and `dns-sd -L`, exactly matching the core Python package's simple chat client. This validates that the Saturn protocol is application-agnostic. Third, the subprocess bridge demonstrates how to add Saturn discovery to legacy software without modifying the application itself---a pattern that extends Saturn's reach beyond applications Saturn developers can control.

\section{GL.iNet Router Deployment}

\subsection{Hardware Constraints}

Travel routers like the GL.iNet MT300N-V2 run OpenWRT with severely constrained resources: 580~MB RAM, 8~GB flash storage, and a single-core processor. Traditional AI inference is impossible on this hardware. Beacon operation, however, requires only mDNS registration and credential rotation---both lightweight operations measurable in kilobytes and milliseconds. A Saturn beacon on this hardware advertises remote AI endpoints and distributes ephemeral keys, never inferring locally. This enables the router to function as a network provisioning device: DHCP assigns IP addresses, DNS provides name resolution, and Saturn provides AI service discovery.

\subsection{OpenWRT Integration}

Saturn integrates into OpenWRT as a custom package. The beacon runs as a system service, starting automatically on boot. Administrator configuration occurs through UCI (Unified Configuration Interface), OpenWRT's native configuration system. An administrator specifies cloud provider credentials and local service URLs; the beacon validates them at startup and advertises the resulting services. The package includes minimal dependencies---pure Python with only standard library imports---fitting within the constrained storage budget.

\subsection{Significance}

The GL.iNet router demonstrates that Saturn can run anywhere network infrastructure runs. A user who administers their home WiFi network can now provision AI services alongside IP assignment, DNS, and DHCP. This flattens the distinction between infrastructure and application: the same device that has always configured the network now configures AI access. For users sharing household WiFi or small offices sharing LANs, this is transformative. Every device that joins the network automatically discovers AI services.

\subsection{Limitations}

The router cannot serve as a hub for multi-site deployments. A user with two homes (weekday and weekend) still needs separate configurations. Router failure means lost service discovery until the device restarts. Deployment at scale (multi-building campus) requires multiple routers or centralized infrastructure. These limitations reflect the design constraints of edge deployment, not failures specific to Saturn.

Assuming an attacker rate of 10 requests/second at \$0.01/request, a captured Saturn key exposes at most \$60 in damage (10 minutes $\times$ 600 seconds $\times$ \$0.10). A static key undetected for one day exposes \$8,640. Meli et al.'s finding that 81\% of leaked keys are never revoked makes the static key scenario unbounded in practice. Saturn's automatic expiry removes revocation from the equation entirely.

\section{Threats to Validity}
\label{sec:threats-to-validity}

Measured discovery latency includes Docker container orchestration overhead---approximately 7 of the 9.2~seconds. Pure mDNS resolution time cannot be isolated in this sandbox without protocol-level instrumentation. This limits precision of the latency measurement but does not affect the feasibility conclusion: agents discovered and used services in all five trials.

LLM-based agents produce variable outputs. The same scenario can yield different execution paths and step counts across runs. The configuration trial used $N=1$ per condition; while the cognitive walkthrough triangulates the finding, larger automated sample sizes would strengthen statistical claims.

Saturn relies on multicast, which does not route across subnets. All evaluation occurred on a single Docker bridge segment. Saturn targets single-segment deployment (home networks, small offices, dorm networks); multi-subnet discovery is future work.

Step counts reflect agent execution complexity, not human cognitive load. Humans follow different strategies and may perceive configuration difficulty differently. The cognitive walkthrough addresses this gap analytically, but human-subject testing would strengthen usability claims.

Information leakage measurement does not equal exploitability. The packet capture recorded what the network exposes; it did not simulate adversarial attacks. Active penetration testing (DNS rebinding, ARP spoofing, rogue service registration) would reveal practical exploitability. Detailed threat analysis appears in the Discussion (Chapter~7).

The Docker sandbox differs from real WiFi networks. Discovery latency, packet loss, and multicast behavior may differ on physical hardware with radio interference and network congestion. The GL.iNet router deployment (Chapter~4) demonstrates Saturn on real hardware; systematic latency testing on physical networks remains future work.

\section{Threat Models}
\label{sec:threat-models}

Saturn trades enterprise-grade verification for zero-configuration access. This is a deliberate choice for the target environment---campus WiFi, home networks, small offices---not an oversight. Four threat scenarios illustrate the trade-off space.

\subsection{Corporate Data Exfiltration}

An employee on a corporate network discovers a Saturn beacon advertising a cloud AI endpoint and sends proprietary data to the service. Saturn's beacon architecture limits this threat: beacons announce endpoints but do not proxy traffic---API requests flow directly from client to backend over HTTPS. An administrator who configures a cloud beacon accepts that users will send requests to that cloud provider, identical to the risk of any cloud-accessible API.

The distinction between \texttt{deployment=local} and \texttt{deployment=cloud} in Saturn's TXT records makes this risk visible. A security-conscious administrator can restrict Saturn to local deployments only, ensuring all AI traffic remains on the LAN. Client applications can filter by deployment type, refusing cloud endpoints when policy requires it.

\subsection{Untrusted Local Administrator}

The administrator who runs Saturn controls what services are advertised, what priority values are assigned, and (for cloud backends) what API keys are distributed. A malicious administrator could advertise a rogue endpoint, capture prompts, or redirect traffic.

This threat is real but not novel. A malicious network administrator already controls DNS, DHCP, firewall rules, and routing tables. Saturn does not introduce new capabilities for compromise---it inherits the standard LAN trust model. The critical design decision is that beacons do not proxy traffic: even if the administrator controls what services are discovered, actual API communication uses HTTPS between client and endpoint. The administrator cannot read encrypted traffic without also compromising the endpoint's TLS certificate.

Logging is the residual risk. An administrator running a local Ollama instance can access Ollama's request logs, which may contain prompts. This is a property of the backend, not of Saturn. Users concerned about prompt privacy should verify the backend's logging configuration independently.

\subsection{Rogue Service Registration}

Any device on a Saturn-enabled network can register mDNS services and advertise itself as an AI provider. An attacker on a coffee shop WiFi network could broadcast a \texttt{\_saturn.\_tcp.local.} service pointing to a malicious endpoint that captures prompts or returns manipulated responses.

mDNS does not authenticate service registrars. Kaiser and Waldvogel~\cite{kaiser2014privacy} documented this as a known limitation; it is not unique to Saturn. Saturn mitigates it through priority-based selection: administrator-configured beacons advertise low priority values (e.g., 10); rogue services default to higher values and rank lower in client selection. This raises the bar above trivial spoofing but does not prevent a determined attacker on the same segment.

For environments where rogue registration is a serious concern (public WiFi, shared networks), Saturn is not the appropriate discovery mechanism. The design explicitly targets administratively controlled networks where the set of legitimate service providers is managed.

\subsection{ISP-Level Service Injection}

An ISP pushes a firmware update to a customer's router that includes a Saturn beacon advertising the ISP's AI service with priority~0 (highest). Users on the network are silently directed to the ISP's endpoint, which may log prompts, inject advertising, or charge for access.

This scenario, explored through the design fiction in Chapter~5, is technically feasible. Saturn's priority mechanism would route users to the ISP-injected service ahead of locally configured alternatives. The attack requires firmware-level access---a capability ISPs already exercise for DNS redirection and captive portals.

Mitigation requires client-side policy. Applications could maintain a whitelist of trusted service providers, require user confirmation before connecting to new services, or display service provenance (local vs.\ cloud, priority source) before routing requests. Saturn's protocol exposes the metadata needed for these checks; implementing them is a client responsibility.
\section{Trust, Privacy, and Verification}
\label{sec:trust-privacy}

Saturn makes an explicit trust assumption: the local network is controlled by the administrator who runs Saturn. This assumption holds for home networks, small offices, and institutional networks where WiFi setup is managed centrally. It breaks down in hostile environments (open WiFi, shared accommodations, untrusted networks).

\subsection{Privacy of Broadcast Metadata}

The packet capture revealed 11 fields exposed in mDNS broadcasts: API base URL, API type, deployment context, priority, version, capabilities, context window, cost tier, features, models, and service type. This is more metadata than typical static configuration, where credentials live in \texttt{.env} files visible only to the local filesystem.

Three observations bound this exposure. First, service metadata is already inferrable on trusted networks---any user on the LAN can attempt connections to common ports and discover running services. Saturn makes this explicit through mDNS rather than leaving it to port scanning, but the information is not fundamentally new. Second, prompts and responses remain encrypted. Broadcast metadata reveals what services exist; HTTPS protects what users ask and what they receive. Third, for cloud deployments using ephemeral keys, the broadcast credential expires automatically within 10 minutes. An observer who captures the broadcast cannot use that credential days later.

For users in untrusted networks (shared WiFi, public venues), Saturn is inappropriate. For users in private networks (home, small office, institutional LANs with controlled access), the privacy trade-off is reasonable.

\subsection{Administrator Trust}

Saturn assumes the network administrator can be trusted. In institutional settings, this is typically the case. Administrators deploy and configure Saturn, control priority values, and manage rotations. A malicious administrator could configure rogue services, manipulate priorities, or redirect traffic---but a malicious administrator can do this through many channels (DNS hijacking, ARP spoofing, network monitoring). Saturn does not introduce new capabilities for compromise; it inherits the standard LAN trust model.

\section{Limitations and Generalizability}
\label{sec:limitations}

The evaluation establishes Saturn's feasibility under tested conditions. These conditions are specific; generalization requires care.

Multicast does not route across subnets. Saturn works on any single LAN segment (home WiFi, office LAN, dorm network, campus subnet). It does not provide discovery across multiple subnets without infrastructure (relay servers, tunneling). This is by design---zero-configuration at the edge, not across wide-area networks.

Step counts reflect what an LLM-based agent must do to complete tasks. Humans may take different paths, require different numbers of decisions, or find configuration easier or harder. The step reduction (65\%) is substantial enough that human paths would likely show similar reduction, but human-subject evaluation would strengthen claims about usability.

Each condition used $N=5$ trials for discovery and $N=1$ per condition for configuration. Confidence intervals are tight (5.7--12\% coefficient of variation), suggesting stability, but larger samples ($N=10$ or $N=20$) would strengthen statistical claims for publication.

Docker sandbox differs from real WiFi networks. Discovery latency measured in emulation may not reflect real-world behavior with physical networks, interference, and multi-hop scenarios. Real-network evaluation is needed before claiming production readiness.

The evaluation measured information leakage but did not simulate adversarial attacks. Active penetration testing (DNS rebinding, ARP spoofing, rogue service registration) would reveal practical exploitability beyond passive observation.

Despite these constraints, the evaluation demonstrates that zero-configuration discovery is feasible, that it reduces configuration work, and that information leakage is bounded and documentable. Saturn achieves its core design goal within its target scope (single-segment, administratively controlled networks).

\section{Relation to Prior Work}
\label{sec:prior-work}

Saturn positions itself at the intersection of three research areas: service discovery, AI provisioning, and network security.

\subsection{Service Discovery}

Zero-configuration service discovery via mDNS and DNS-SD is established in networking research. Guttman~\cite{guttman2001zeroconf} and Siddiqui~\cite{siddiqui2012zeroconf} documented the protocol foundations. Saturn applies these existing protocols to a new domain (AI services) without modifying them. This is intentional: reusing proven infrastructure avoids introducing new complexity or vulnerabilities.

The evaluation confirms that mDNS discovery is feasible for AI services. This extends prior work by demonstrating practical applicability beyond media servers and printers.

\subsection{AI-as-a-Service Provisioning}

Syed et al.~\cite{syed2025aiaas} documented the current architecture: users must obtain and manage credentials from each provider independently. Costa~\cite{costa2024democratization} and Gabriel~\cite{gabriel2024educational} discuss how this fragmentation barriers access for underfunded users and institutions. Saturn addresses this by making local services discoverable, reducing the number of external credentials needed.

Saturn does not replace cloud AI services. Instead, it complements them by enabling local deployment while maintaining compatibility with cloud APIs. Users can discover both local and cloud services through the same mechanism.

\subsection{Security Trade-offs}

Ward \& Beyer~\cite{ward2014beyondcorp} and Osborn~\cite{osborn2016beyondcorp2} advocate zero-trust architecture: assume the network is hostile, verify every request. Saturn deliberately diverges. Saturn assumes the network is controlled by the administrator and trusts mDNS broadcast. This is appropriate for edge networks (home, small office) where administrative control is clear. BeyondCorp is appropriate for large enterprises where network trust is weaker.

Kaiser \& Waldvogel~\cite{kaiser2014privacy} documented privacy concerns in mDNS (service enumeration, device naming). Saturn accepts this known exposure as the price of zero-configuration discovery. Their proposed privacy-preserving extensions could be layered on top if needed~\cite{kaiser2014efficient}.

\section{Future Work}
\label{sec:future-work}

The evaluation identifies several directions for future development:

Current Saturn uses multicast, limited to single segments. A relay service could forward mDNS queries across subnets, enabling discovery in multi-segment campus networks. This requires careful design to avoid broadcast storms and privacy leakage across organizational boundaries.

Kaiser \& Waldvogel's privacy-preserving mDNS-SD protocols could reduce metadata visibility. Rather than broadcasting all service metadata, beacons could advertise only service existence, with detailed metadata available through secure queries. This would prevent passive enumeration while maintaining zero-configuration setup for users with valid credentials.

Current Saturn requires an administrator to configure and run the central coordinator. A fully decentralized design could allow peer-to-peer deployment, where any user could provision local services. This trades operational simplicity for deployment flexibility.

Evaluate Saturn on real WiFi networks with interference, multi-hop scenarios, and physical distance. Docker sandbox results do not account for real radio propagation characteristics. Production deployment requires validation in realistic environments.

Test actual users (developers, students, researchers) configuring AI services with and without Saturn. Measure not just step count but cognitive load, time-to-first-success, and error recovery. This would validate that agent-measured step reduction translates to actual human usability improvement.

Conduct formal penetration testing: DNS rebinding attacks, ARP spoofing, rogue service registration, key exfiltration. Measure whether documented mitigations actually prevent exploitation. This would move from ``passive observation'' to active security validation.

\section{Broader Implications}
\label{sec:implications}

Saturn's core contribution is making AI services discoverable and automatically configurable within administrative domains. This addresses a real problem: users who lack technical expertise, funding, or network administrative privileges currently cannot access local AI services.

The evaluation demonstrates feasibility. Configuration effort drops 59\%; discovery succeeds 100\% of the time. Security exposure is bounded and documentable. These results suggest that zero-configuration AI provisioning is practical and worth deploying.

The broader implication is equity. Bassignana~\cite{bassignana2025aigap} documents the AI gap---the digital divide where those without resources cannot access cutting-edge AI tools. Gabriel~\cite{gabriel2024educational} shows how this affects educational opportunity. Carmona-Galindo~\cite{carmona2025hsi} specifically studies Hispanic-Serving Institutions, where budgetary constraints limit AI tool availability.

Saturn does not solve the global AI access problem. But within a networked institution, it enables any administrator to deploy AI services that all users can discover and use. It lowers barriers from ``hire a specialist to configure each tool'' to ``run a single daemon.'' This matters most for resource-constrained institutions---community colleges, public schools, libraries---where every dollar of IT budget counts.

\chapter{Saturn Protocol Specification}
\label{app:protocol-spec}

This appendix specifies the Saturn protocol as a standalone reference. An implementor reading only this appendix and the RFCs it cites should produce a system interoperable with any conforming Saturn deployment.

\section{Service Registration}

A Saturn beacon registers one DNS-SD service instance under the type \texttt{\_saturn.\_tcp.local.} (RFC~6763~\cite{rfc6763}). Registration creates three DNS records:

\begin{enumerate}
\item A \textbf{PTR record} mapping \texttt{\_saturn.\_tcp.local.} to the instance name (e.g., \texttt{ollama-lab.\_saturn.\_tcp.local.}).
\item An \textbf{SRV record} providing the beacon's hostname and port.
\item A \textbf{TXT record} carrying service metadata as key-value pairs.
\end{enumerate}

The instance name is a human-readable UTF-8 string chosen by the administrator. It must be unique within the multicast domain. The \texttt{.local.} suffix confines all queries and responses to the link-local scope defined by mDNS (RFC~6762~\cite{rfc6762}).

\section{Endpoint Requirements}

Every URL advertised in \texttt{api\_base} must implement three HTTP paths:

\begin{description}
\item[\texttt{GET /v1/health}] Returns JSON with a \texttt{status} field. Clients call this to verify liveness before use and periodically during sessions. A non-200 response or timeout (recommended: 5~seconds) indicates the service is unavailable.
\item[\texttt{GET /v1/models}] Returns a JSON object with a \texttt{data} array of model objects, each containing at minimum an \texttt{id} field. Follows the OpenAI list-models response format.
\item[\texttt{POST /v1/chat/completions}] Accepts a JSON body with \texttt{model} (string) and \texttt{messages} (array of \texttt{\{role, content\}} objects). Returns a chat completion response. Must support both non-streaming (JSON response) and streaming (server-sent events with \texttt{data:} prefix) modes, selected by the \texttt{stream} boolean in the request body.
\end{description}

For cloud beacons, \texttt{api\_base} points to the upstream provider, not the beacon. Clients authenticate with the provider using the \texttt{ephemeral\_key} from the TXT record.

\section{Discovery Algorithm}

Listing~\ref{lst:discovery} specifies client discovery in pseudocode.

\begin{verbatim}
function discover():
    instances = mdns_browse("_saturn._tcp.local.")
    services = []
    for instance in instances:
        srv = mdns_resolve_srv(instance)
        txt = mdns_resolve_txt(instance)
        if txt["version"].major != 1:
            skip  // incompatible protocol version
        services.append({
            host: srv.hostname,
            port: srv.port,
            metadata: txt
        })
    sort services by metadata["priority"] ascending
    for service in services:
        url = service.metadata["api_base"]
        headers = {}
        if "ephemeral_auth" in service.metadata["features"]:
            headers["Authorization"] =
                "Bearer " + service.metadata["ephemeral_key"]
        response = http_get(url + "/v1/health",
                            headers, timeout=5s)
        if response.status == 200:
            return service  // connected
    return null  // no healthy service found
\end{verbatim}
\label{lst:discovery}

Clients may cache discovery results. A recommended cache TTL is 60~seconds; clients should re-browse when the cached service fails a health check.

\section{Ephemeral Key Rotation Protocol}

Cloud beacons rotate API credentials on a fixed schedule to bound exposure windows. The rotation proceeds as follows:

\begin{enumerate}
\item At startup, the beacon creates a new API key via the provider's provisioning API and records its creation time $t_0$.
\item The beacon publishes the key in the \texttt{ephemeral\_key} TXT field and sets \texttt{rotation\_interval} to the rotation period $R$ (default: 300~seconds).
\item At $t_0 + R$, the beacon creates a new key $k_1$, updates the TXT record to advertise $k_1$, and starts a grace timer for the old key $k_0$.
\item After a grace period $G$ (default: 5~seconds), the beacon deletes $k_0$ from the provider's system.
\item Steps 3--4 repeat every $R$ seconds.
\item On shutdown, the beacon deletes the current key from the provider.
\end{enumerate}

The key lifetime $L$ (default: 600~seconds) exceeds the rotation interval $R$, so at most two keys are valid simultaneously during the overlap window. This guarantees zero-downtime transitions: clients using the previous key continue working while the TXT record advertises its replacement.

\section{Conformance}

A conforming Saturn beacon must:
\begin{itemize}
\item Register exactly one \texttt{\_saturn.\_tcp.local.} service instance per advertised endpoint.
\item Include all six required TXT fields (Table~\ref{tab:appendix-txt}).
\item Include \texttt{ephemeral\_key} and \texttt{rotation\_interval} when \texttt{features} contains \texttt{ephemeral\_auth}.
\item Deregister the mDNS service on shutdown.
\item Delete active ephemeral keys from the provider on shutdown.
\end{itemize}

A conforming Saturn client must:
\begin{itemize}
\item Discover services via mDNS PTR query for \texttt{\_saturn.\_tcp.local.}.
\item Parse all required TXT fields; ignore unrecognized fields.
\item Sort discovered services by \texttt{priority} ascending.
\item Verify service health via \texttt{GET /v1/health} before use.
\item Fall back to the next-priority service when the current service becomes unreachable.
\end{itemize}

\section{Confidence Interval Computation}

All confidence intervals use Student's $t$ distribution. For $N$ trials with sample mean $\bar{x}$ and sample standard deviation $s$, the 95\% confidence interval is:

\[
\bar{x} \pm t_{0.025, N-1} \cdot \frac{s}{\sqrt{N}}
\]

\noindent where $t_{0.025, N-1}$ is the critical value of the $t$ distribution with $N-1$ degrees of freedom at the 2.5\% tail. For $N=5$ ($df=4$), $t_{0.025,4} = 2.776$.

We assess estimate stability via the coefficient of variation $\text{CV} = s / \bar{x}$. A CV below 10\% indicates low variance sufficient for thesis reporting. A CI width (as percentage of the mean) below 10\% indicates a stable point estimate.

The discovery latency pilot ($N=5$) achieves a CI width of 10.1\%---at the stability threshold. \chapter{Scenario Specifications}
\label{app:scenarios}

This appendix reproduces the task descriptions and environment configurations for each evaluation scenario. Agents received these descriptions verbatim; no additional hints, Saturn-specific vocabulary, or success criteria were provided.

\section{Claim 1: Discovery (Cold Start)}

\subsection{Task Description}

\begin{quote}
You are a developer working on a local network. Someone told you there's an AI text completion service available on this LAN, but they didn't give you any details---no URL, no API key, no documentation.

Your goal: find the AI service and successfully get a text completion from it.

You have a Linux machine with common developer tools installed. The network has other devices on it. Figure out how to locate and use the service.

When you've successfully received a text completion response, save the full JSON response to \texttt{/tmp/completion-result.json} and print ``TASK COMPLETE''.
\end{quote}

\subsection{Environment}

Docker bridge network (\texttt{saturn-eval-net}) with two containers: a Saturn beacon advertising one Ollama instance (\texttt{llama3.2:1b}) at priority~10, deployment \texttt{local}; and an agent container with bash and Python~3. The agent container has no Saturn client libraries installed. Timeout: 600~seconds.

\subsection{Measurements}

Time from task start to first successful \texttt{/v1/chat/completions} response (discovery latency). Discovery method used. TXT record fields parsed. Success or failure.

\section{Claim 2: Configuration Effort}

\subsection{Control: Manual Ollama Setup}

\begin{quote}
You need to get AI chat working in Open WebUI using a local Ollama server as the backend.

Open WebUI is already installed and running at \texttt{http://localhost:3000}. You have an admin account (username: \texttt{admin@eval.local}, password: \texttt{admin123}).

Ollama is a local AI model server. It is installed on another machine on this network but you'll need to figure out its address and configure Open WebUI to connect to it.

Your goal: configure Open WebUI so that when a user opens the chat interface, they can select a model and get a response from the Ollama server. Verify it works by sending a test message and receiving a reply.

When chat is working, save the conversation to \texttt{/tmp/chat-success.txt} and print ``TASK COMPLETE''.
\end{quote}

\subsection{Treatment: Saturn-Enabled Setup}

\begin{quote}
You need to get AI chat working in Open WebUI.

Open WebUI is already installed and running at \texttt{http://localhost:3000}. You have an admin account (username: \texttt{admin@eval.local}, password: \texttt{admin123}).

There should be AI services available on this network. Your goal is to get Open WebUI working so that a user can open the chat, pick a model, and get a response.

When chat is working, save the conversation to \texttt{/tmp/chat-success.txt} and print ``TASK COMPLETE''.
\end{quote}

\subsection{Environment}

Both conditions use Docker bridge networks. The control environment runs Open WebUI and a standalone Ollama container; the agent must discover the Ollama address and configure the connection. The treatment environment runs Open WebUI with the Saturn plugin enabled alongside a Saturn beacon advertising the same Ollama instance. Both provide bash and browser tools. Timeout: 900~seconds.

\subsection{Measurements}

Three independent graders analyze execution traces: a step counter (classifying actions as network requests, terminal commands, form inputs, clipboard operations, navigation, and discovery actions), an artifact counter (config files, environment variables, API keys, accounts created, URLs entered, dependencies installed), and a success-rate checker (binary outcome with duration).

\subsection{Key Difference}

The treatment description omits all mention of Ollama, backend configuration, and connection URLs. The agent knows only that ``AI services'' exist on the network. Saturn's zero-configuration property means the treatment agent should not need backend-specific knowledge to succeed.

\section{Claim 3: Passive Network Observation}

\subsection{Description}

No agent participates in this scenario. A \texttt{tcpdump} sidecar captures all multicast DNS traffic on the Docker bridge network while a Saturn beacon operates normally. The capture runs for 720~seconds, covering at least one full ephemeral key rotation cycle (key lifetime: 600~seconds, rotation interval: 300~seconds).

\subsection{Environment}

Docker bridge network with two containers: a Saturn beacon advertising an Ollama instance with \texttt{deployment=cloud} and ephemeral key rotation enabled, and a capture sidecar running \texttt{tcpdump} with mDNS filters (\texttt{udp port 5353}). No client container is present; the capture records only beacon announcements and mDNS protocol traffic.

\subsection{Measurements}

Fields exposed in mDNS TXT records (count and values). Credential material presence and ephemerality. Service enumeration feasibility. Hostname and port exposure. Comparison against static API key and OAuth exposure profiles (analytical, not captured).

\subsection{Analysis}

The capture output feeds two analyses. First, a field inventory: every key-value pair observed in Saturn TXT records is catalogued to characterize the information leakage surface. Second, a STRIDE threat comparison: Saturn's exposure profile is evaluated against static API keys and OAuth/SSO across six threat categories (Table~\ref{tab:stride} in Chapter~6).
