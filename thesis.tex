\vspace{0.5in}

AI tools gate access behind API keys. Each application needs its own key, its own endpoint, its own setup---three tools, three accounts. A student without a credit card gets zero. Saturn sidesteps the whole ritual by broadcasting AI endpoints over mDNS and DNS-SD, the protocols behind AirPrint and Chromecast, so that joining a WiFi network is enough to reach a language model.

Saturn beacons register OpenAI-compatible endpoints under \texttt{\_saturn.\_tcp.local.} and publish metadata plus a short-lived credential in DNS TXT records. Keys rotate every five minutes; grab one off the wire and it dies before you leave the building. Clients discover beacons, rank them by administrator-set priority, and connect---no accounts, no credential files. One network administrator replaces per-user key management for every device on the segment. The security exposure is bounded: Saturn broadcasts only what fits in a DNS TXT record, and ephemeral rotation makes captured keys worthless within minutes.

We built six implementations in four languages. One runs on a GL.iNet travel router next to the DHCP daemon---same box, same boot, but now it hands out AI access alongside IP addresses. mDNS has handled printers since 2002. Saturn puts language models on the same wire.
}

\chapter{Introduction}
% ---------------------------------------------------------\section{The Problem: AI Access Inequity}

Lewis Strauss predicted in 1954 that atomic energy would become ``too cheap to meter''~\cite{strauss1954}. Generative AI landed with the same promise and the same fine print. ChatGPT Plus costs \$20/month~\cite{chatgpt-plus}; Claude Pro matches it~\cite{claude-pricing}. A researcher who subscribes to both pays \$40 before usage-based tiers push the bill past \$100.

Price is the visible cost. Underneath sits a structural one. Syed et al.~\cite{syed2025aiaas} documented the architecture: REST endpoints gated by per-user tokens, each obtained and rotated independently. A developer running chat, code completion, and transcription against a single Ollama instance types the address three times. Change the port and all three break.

Those credentials leak. Meli et al.~\cite{meli2019secrets} scanned GitHub and turned up API secrets in over 100,000 repositories; they notified the owners and 81\% of the keys stayed live. Static keys---still the default at every major AI provider---are the worst case: three quarters of those Meli's scanner flagged sat unrevoked.

Who absorbs this friction depends on who they already are. Bassignana et al.~\cite{bassignana2025aigap} found that socioeconomic status predicts how much value a person extracts from language technology even when access is nominally free. Gabriel and Bossens~\cite{gabriel2024educational} traced the same dynamic through classrooms; Capraro et al.~\cite{capraro2024impact} measured it across national economies. Costa et al.~\cite{costa2024democratization} named the result: technical barriers reinforcing the power asymmetries they were supposed to level.

\section{The Vision: AI as Network Infrastructure}

Apple shipped Bonjour in 2002. Within a year, printers appeared on Macs without drivers or IP addresses. The mechanism: Multicast DNS (mDNS) resolves device names on the local segment without a central server, and DNS-based Service Discovery (DNS-SD) advertises what each device offers. Guttman~\cite{guttman2001zeroconf} had described the aspiration---``direct communications between two or more computing devices via IP'' with no manual setup. RFC~1122~\cite{rfc1122} put it more bluntly: ``It would be ideal if a host implementation of the Internet protocol suite could be entirely self-configuring.''

Language models missed this transition. Every chat application, every coding assistant, every transcription tool asks the same questions on first launch: which provider? which key? which endpoint? An address, a credential, a protocol dialect---exactly the information printers have broadcast since Bonjour shipped.

The precedent runs deeper than desktops. Eduroam provisions WiFi credentials across hundreds of campuses; campus print services discover printers without configuration files. Saturn applies the same principle to language models: join a network and the AI services on it appear.

\section{Thesis Statement}

Generative AI should be provisioned the way printers and speakers already are: at the network level. Saturn broadcasts where AI services live and how to authenticate using mDNS and DNS-SD---protocols shipping on billions of devices already. One administrator configures the network; every other user and application inherits the result. The cost of that convenience is visibility: every device on the segment sees service metadata, including short-lived credentials that expire within minutes.

\section{Saturn and This Thesis}

Under \texttt{\_saturn.\_tcp.local.}, servers register OpenAI-compatible endpoints; clients browse the service type and connect without setup. The protocol separates announcement from traffic: beacons broadcast metadata---where services live, how to authenticate---but never proxy API calls. A crashed beacon leaves active connections untouched. Everything a client needs---endpoint URL, API type, credentials, routing priority---fits inside standard DNS-SD TXT records, and no side channel carries configuration.

We built six consumers to test whether the specification, not the reference code, carries the design. A core Python package, a TypeScript provider for Vercel's AI SDK, an Open WebUI plugin, an MCP server for coding assistants, a Lua-to-Python bridge inside VLC, and a fork of the OpenCode terminal. Four languages, four mDNS libraries, no shared discovery code. Each is short enough for a developer to read end-to-end.

The seventh component is a Rust binary cross-compiled to MIPS32 and running on a \$30 GL.iNet travel router---the same box that assigns IP addresses via DHCP and resolves names via DNS. Saturn sits in its process table alongside those services, managed by the same init system, configured through the same web interface. That deployment moves the feasibility argument off developer workstations and onto consumer hardware.

Chapter~2 surveys existing service-discovery protocols and the AI applications Saturn targets. Chapter~3 specifies the protocol; Chapter~4 walks through each implementation.

Claude, Anthropic's language model, helped build the knowledge graph behind this thesis and gave feedback on drafts. Saturn argues AI should be accessible; concealing our own use of it would undercut the argument. Appendix~B details what Claude contributed and where we overruled it.

\chapter{Background}
% ------------------------------------------------------------

A laptop and a printer share a switch and cannot see each other. For three decades, competing protocols have tried to solve this---making devices visible without manual intervention. None were designed for AI. This chapter explains why mDNS and DNS-SD fit Saturn's requirements where the alternatives fall short, then surveys the AI applications Saturn targets.

\section{Service Discovery on Local Networks}

Plug a laptop into a network and the printer three feet away is invisible. Service discovery is the fix: protocols that let devices announce themselves and clients find what is available.

\subsection{NetBIOS}

Network Basic Input/Output System (NetBIOS) resolved names on Windows local networks before DNS displaced it~\cite{rfc1001}. macOS and Linux never adopted it, Microsoft began phasing it out with Windows~10, and the protocol carries no service metadata---it cannot advertise model lists, API types, or capabilities.

\subsection{Universal Plug and Play}

UPnP's security record is disqualifying: remote code execution, unauthorized port forwarding, and a history of CVEs that stretches back two decades~\cite{upnp2015}. All three major OS vendors have been pulling functionality toward mDNS/DNS-SD instead. UPnP layers SOAP/XML control on top of SSDP---machinery built for controlling devices, not for advertising metadata.

\subsection{DLNA}

The Digital Living Network Alliance dissolved in 2017~\cite{dlna2017}. Before that, DLNA built a media streaming standard on UPnP so televisions could find content servers. The scope never extended beyond audio and video.

\subsection{WS-Discovery}

Web Services Dynamic Discovery (WS-Discovery) targets enterprise environments with managed infrastructure~\cite{wsdiscovery}. Windows uses it for printer discovery alongside mDNS. The SOAP/XML encoding is heavier than Saturn needs and the library ecosystem is sparse, but the fundamental mismatch is the network assumption: WS-Discovery expects IT departments. Saturn targets places without them.

\subsection{Why mDNS/DNS-SD}

mDNS resolves names without a server. DNS-SD advertises services using standard DNS record types. Both run on every major OS---under AirPrint, AirPlay, Chromecast, Spotify Connect---on billions of devices. Guttman~\cite{guttman2001zeroconf} defined zero-configuration networking as ``direct communications between two or more computing devices via IP'' without manual setup. Nothing in production comes closer. Section~2.2 covers both protocols.

\subsection{DHCP}

DHCP is Saturn's closest analogy at the network layer~\cite{rfc2131}. A device broadcasts DISCOVER; a DHCP server replies with an IP address, gateway, DNS server, and lease duration; the client acknowledges and is online. No manual configuration. Saturn's ``join the network, get AI'' pitch is consciously modeled on DHCP's ``join the network, get an IP.''

The analogy has limits. DHCP fires once at connection time, assigns one address, and requires root privileges. Saturn runs continuously in userspace because AI services appear and disappear---a local Ollama instance spins up, a cloud key rotates, a beacon goes offline. A network might host a dozen services with different models, priorities, and credential types. DHCP carries a handful of fields in its options; Saturn carries model lists, API types, ephemeral keys, and rotation intervals in DNS TXT records.

They complement each other. On Saturn's GL.iNet router, DHCP assigns IP addresses while Saturn advertises AI endpoints. Same hardware, same boot sequence, different layers.

\section{mDNS and DNS-SD}
\label{sec:mdns-dnssd}

\subsection{Multicast DNS}
\label{sec:mdns-detail}

RFC~6762~\cite{rfc6762} specifies Multicast DNS (mDNS). Instead of querying a designated resolver, the client multicasts to 224.0.0.251:5353 (ff02::fb on IPv6). Every device on the link hears it. Whichever device owns the name responds with its IP address. No server, no authority, no configuration.

Names end in \texttt{.local}---\texttt{my-server.local}, say. The wire format is ordinary DNS: A, AAAA, SRV, TXT, PTR. Only two things differ from conventional DNS: transport (multicast instead of unicast) and scope (link-local instead of global). Siddiqui et al.~\cite{siddiqui2012zeroconf} place mDNS alongside link-local addressing~\cite{rfc3927} and DNS-SD as the three pillars of zero-configuration networking.

\subsection{DNS-Based Service Discovery}
\label{sec:dnssd-detail}

mDNS resolves names. DNS-based Service Discovery (DNS-SD, RFC~6763~\cite{rfc6763}) answers the next question: what services of type X exist on this network?

Three record types chain together. PTR maps a service type---\texttt{\_saturn.\_tcp.local.}---to instance names. SRV gives hostname and port for each instance. TXT carries key-value metadata. A client queries PTR, gets instance names back, then resolves SRV and TXT for each.

DNS-SD rides on mDNS for link-local scope or on unicast DNS for wide-area. Saturn uses mDNS only, confining discovery to the local segment.

\subsection{TXT Records and Metadata}

TXT records carry key-value pairs. The DNS wire format caps each string at 255~bytes; a record can hold multiple strings, each independently limited.

Saturn uses eight TXT fields: protocol version, deployment type, API type, base URL, priority, ephemeral key, rotation interval, feature flags. A client that resolves these knows enough to connect---no follow-up needed. The 255-byte ceiling constrains field sizes (an ephemeral key gets roughly 240 usable characters after accounting for the key name and delimiter), but Saturn's metadata fits.

\subsection{Implementations: Bonjour and Avahi}

Bonjour shipped with macOS in 2002 and powers AirPrint, AirPlay, and AirDrop~\cite{bonjour}. Linux distributions run Avahi~\cite{avahi}. Microsoft added a native mDNS responder in Windows~10. Every major desktop and mobile OS already speaks mDNS. An application that calls into the system's resolver---any language, any framework---can discover Saturn services without installing anything.

\subsection{Adoption in New Domains}

Printers and file shares came first, then AirPlay for speakers, Chromecast for media casting, Spotify Connect for playback. Siljanovski et al.~\cite{siljanovski2014mdns} extended mDNS into IoT sensor networks; VS~Code uses it to locate remote debugging targets without IP addresses~\cite{vscode-remote}.

Saturn's most direct precedent is a community pull request that would have added mDNS advertisement to Ollama~\cite{ollama-mdns-pr}. It was never merged. But the impulse behind it---AI services announcing themselves the way printers do---is exactly Saturn's starting point. \texttt{\_saturn.\_tcp} instead of \texttt{\_ipp.\_tcp}, same machinery underneath.

\subsection{Privacy Considerations}

mDNS queries and responses reach every device on the link. Kaiser and Waldvogel~\cite{kaiser2014privacy} showed that a passive observer can enumerate services, track devices, and correlate metadata with users. K\"{o}nings et al.~\cite{konings2013devicenames} found that 59\% of device names in the wild contain real user names---mDNS announcements as PII.

Saturn inherits all of it. Anyone on the segment sees which AI services exist, which models they serve, and---depending on beacon naming---who operates them. Kaiser and Waldvogel~\cite{kaiser2014efficient} proposed hash-based instance names to limit observer knowledge without breaking discovery; Chapter~7 evaluates Saturn against these known weaknesses.

\section{The AI Service Landscape}

Chat interfaces, coding assistants, voice pipelines. Every application that consumes AI text completion asks the same questions on first launch: where is the endpoint, what are the credentials, which model.

\subsection{Hosted Chat Applications}

Open WebUI~\cite{open-webui}, LibreChat~\cite{librechat}, Jan~\cite{jan-ai}: three chat interfaces, same backend, three separate configurations. Each wants a base URL, an API key, a model identifier. Each stores its own copy. Change the port on the backend and all three break until someone re-enters the address by hand.

\subsection{Coding Agents and Assistants}

Cursor~\cite{cursor} for code completions, Open WebUI for chat---same Ollama backend, configured twice. Add Continue~\cite{continue} and it becomes three. Each tool keeps its own URL and API key. None knows the others exist.

\subsection{Voice Typing and Transcription}

Voice-to-text pipelines pair a speech recognizer (Whisper~\cite{whisper}) with a language model for cleanup. SuperWhisper~\cite{superwhisper} runs both on-device. Wispr Flow~\cite{wisprflow} sends transcripts to a cloud LLM at \$8/month. MacWhisper~\cite{macwhisper} bundles local models. Want local transcription with cloud cleanup? Two models, two credentials, two failure points.

\chapter{Design}
% ------------------------------------------------------------

Add a printer to a network and your laptop finds it. Add an AI service and nothing happens---someone still has to type a URL, paste a key, pick a model, app by app. Saturn registers AI services the same way printers register themselves: mDNS announcement, DNS-SD metadata, zero client configuration.

\section{Goals}

Join a network, get AI. No installation, no accounts, no keys.

The protocol cannot depend on a single vendor. Anthropic could raise prices, OpenRouter could shut down, a university might switch from one provider to another mid-semester. Saturn binds to an interface standard---the OpenAI-compatible REST API---not a company. Any backend behind that API qualifies.

One person sets up services; every device on the segment inherits them. A university sysadmin, a roommate with a GPU, someone sharing a cloud key for the afternoon---the effort scales with the number of services, not the number of people.

Captured credentials are the obvious risk. Someone records a key from a beacon and walks out. So cloud-facing beacons rotate keys every five minutes; a stolen credential dies before you reach a different network.

\section{Audiences}

Configuration labor does not disappear. Saturn concentrates it on one person so everyone else does none.

\textbf{Providers}---OpenAI, Anthropic, OpenRouter---change nothing.

\textbf{Administrators} carry the full weight: obtaining keys, deploying beacons, naming services, setting priorities. The zero-configuration experience exists because this person configured everything in advance.

\textbf{End users} open an application, find AI available, use it. They never see Saturn's name. The closest analogy is AirPlay---join a network, open a compatible app, the service is already there.

\textbf{Developers} write four lines: install, import, \texttt{discover()}, connect. No API key management. No billing integration. No provider selection logic.

\section{Concepts}

Four abstractions carry the protocol.

\subsection{Endpoints}

Ollama, OpenRouter, LiteLLM, vLLM, and dozens of other providers already serve the OpenAI-compatible REST API---JSON messages in, completions out. Saturn adopts that interface wholesale. A Saturn endpoint is any HTTP service behind it: cloud provider, university data center, single-board computer running quantized models. If the service handles the paths in Section~3.4.3, it qualifies. No code changes, no Saturn-specific integration.

\subsection{Beacons}

Beacons do not touch API traffic. A beacon registers a service under \texttt{\_saturn.\_tcp.local.}, publishes metadata in DNS TXT records, and answers discovery queries. Clients connect to the advertised endpoint directly---the beacon says where to go and how to authenticate but never sees a prompt or a response.

That separation has consequences. Kill the beacon and active connections survive; clients already holding an endpoint address keep using it. The beacon's only job is DNS registration and credential management, which is why it runs on hardware as constrained as a travel router (Section~4.4).

Two types exist. A \emph{cloud beacon} announces a remote provider and hands out ephemeral API keys (Section~3.3.4). A \emph{local beacon} announces something on the LAN---an Ollama instance, a vLLM server---and pulls the DNS entry when the service goes down.

\subsection{Priorities}

A network might run a local GPU and keep a cloud API as fallback. Priority---a number in each service's TXT record---determines which wins. Lower beats higher: priority~1 over priority~50. Semantics follow DNS SRV convention.

The administrator who wants local-first inference assigns priority~1 to the GPU, priority~50 to the cloud. Users get whichever is available without choosing or knowing the difference. Priority is policy, not performance---the administrator decides what ``best'' means for this network.

\subsection{Ephemeral Keys}

Broadcasting an API key to every device on the segment hands credentials to anyone listening. Saturn limits the damage through time. Cloud beacons generate keys that live ten minutes on the provider's server and rotate every five. Old and new keys overlap so clients never lose connectivity mid-request. Ten minutes after issuance, the beacon revokes the old key through the provider's API.

Record a credential from a broadcast, walk out of the building, try it from a coffee shop---revoked. The mechanism resembles Kerberos ticket distribution: time-limited credentials scoped to a session, applied here at the application layer through provider key provisioning APIs.

Local deployments skip the mechanism entirely. An Ollama instance on a private network needs no authentication; the TXT record's \texttt{features} field (Section~3.4.2) says so. Cloud deployments require that the provider support programmatic key creation and deletion---OpenRouter and DeepInfra do; providers without provisioning APIs cannot participate.

\section{Protocol Specification}

Combined with RFCs~6762 and~6763, what follows is sufficient to build a conforming implementation from scratch.

\subsection{Service Type}

Saturn services register under the DNS-SD service type \texttt{\_saturn.\_tcp.local.}. The underscore-prefixed format follows RFC~6763 naming conventions for DNS-SD service types~\cite{rfc6763}. The \texttt{\_tcp} designator indicates that clients connect via TCP after discovery. The \texttt{.local.} suffix confines discovery to the link-local multicast domain defined by mDNS~\cite{rfc6762}.

\subsection{Record Structure}

Saturn reuses the DNS-SD record chain from Section~\ref{sec:dnssd-detail}. Three record types cooperate per service instance. A \textbf{PTR record} maps \texttt{\_saturn.\_tcp.local.} to instance names---the enumeration step. An \textbf{SRV record} per instance gives hostname and port. A \textbf{TXT record} per instance carries the rest: deployment type, API compatibility, priority, and---for cloud beacons---an ephemeral credential. Once a client has all three records, it can connect without further queries.

Table~\ref{tab:txt-schema} specifies the TXT record fields that constitute the Saturn protocol.

\begin{table}[ht]

\subsection{Endpoint Expectations}

Three HTTP paths are mandatory (Table~\ref{tab:endpoints}).

Cloud beacons are different: \texttt{api\_base} points to the upstream provider, not to the beacon. Clients extract the \texttt{ephemeral\_key} from the TXT record and authenticate with the provider directly.

\subsection{Discovery Process}

Discovery is four steps:

\textbf{Step~1: Browse.} The client multicasts a DNS-SD PTR query for \texttt{\_saturn.\_tcp.local.}, requesting the names of all Saturn service instances on the local network.

\textbf{Step~2: Resolve.} For each discovered instance, the client queries the corresponding SRV and TXT records. The SRV record yields a hostname and port; the TXT record provides metadata including deployment type, API base URL, priority, and---for cloud beacons---an ephemeral credential.

\textbf{Step~3: Sort.} The client orders discovered services by the \texttt{priority} field in ascending order. Ties may be broken by implementation-defined criteria such as the \texttt{cost} field or random selection.

\textbf{Step~4: Select.} The client issues a \texttt{GET /v1/health} request to the highest-priority service. If the service responds, the client connects. If not, the client proceeds to the next service in priority order. Discovery fails only when no service responds.

If a connected service drops---failed health check or failed API request---the client starts over from Step~1.

\section{Architecture Decisions}

A proxy would be simpler for clients---one URL, one connection---but the proxy sees every prompt and response, fails and takes the whole network with it, and must carry throughput it didn't generate. Beacons sidestep all three. Clients discover endpoints at startup and reconnect on failure; the beacon itself never touches API traffic.

Multicast confines discovery to a single broadcast segment. A centralized registry would cross subnets and look things up faster. It would also need an operator, a domain name, TLS certificates, and an availability plan. mDNS needs the network and nothing else. The scope limitation is deliberate: Saturn targets home networks, campus WiFi, small offices---places where one segment is the whole network.

The specification defines a service type, a TXT record schema, and endpoint contracts. It does not prescribe a language, an mDNS library, or an SDK. Seven implementations in four languages (Section~\ref{sec:r3-interop}) prove the point: interoperability follows from the wire format, not from shared code.

Image generation, audio synthesis, and embedding computation each use different API shapes. Saturn handles text completion only---the capability the OpenAI API standardizes and the one deployed most widely. Other modalities would need their own TXT schema extensions and endpoint contracts, not a redesign of the mDNS/DNS-SD layer.

\subsection{Server Architecture}

On macOS the core Python package talks to Bonjour; on Linux, Avahi; when neither daemon is running, it drops to raw UDP multicast on 224.0.0.251:5353. Beacon registration, TXT record refresh, mDNS TTL management, and credential rotation for cloud deployments all live here.

\subsection{Client Discovery}

Discovery follows the four-step process from Section~3.4.4: \texttt{discover()} multicasts a PTR query for \texttt{\_saturn.\_tcp.local.}, resolves each service's SRV and TXT records, sorts by priority, and health-checks the results. \texttt{connect(service)} reads the ephemeral key from the TXT record and authenticates. A 60-second cache avoids redundant queries; a failed health check forces a fresh browse.

\section{Server Types}

\subsection{Ollama Server}

Port 11434 by default, or whatever the administrator specifies. The beacon confirms the Ollama instance is alive via \texttt{/v1/health}, pulls loaded models from \texttt{/api/tags}, and packs both the endpoint and the model list into a single TXT record. Clients learn what is available at browse time---no follow-up query.

\subsection{OpenRouter Server}

Rotation is the point of the OpenRouter beacon. OpenRouter's provisioning API lets callers create scoped sub-keys with server-side expiry. The beacon authenticates once with the administrator's master key, then mints a fresh 10-minute sub-key every 5 minutes and broadcasts it in the TXT record. When the replacement goes live, the old key is deleted. The administrator's permanent credential never touches the wire.

\subsection{Fallback Server}

The fallback beacon returns canned responses from hardcoded models. It exists for testing: a developer can validate Saturn discovery end-to-end without Ollama running or cloud credentials configured.

Cursor, Continue, and Open WebUI already accept arbitrary OpenAI-compatible endpoints. \subsection{Subprocess Bridge}

Applications that predate Saturn have no mDNS capability. A subprocess bridge works around that: a small local proxy handles discovery on the application's behalf.

VLC media player is the test case. Its users play video files; API keys mean nothing to them. If Saturn can deliver AI inside VLC without any user configuration, the claim extends beyond developer tooling.

\subsubsection{User Experience}

Installation: copy one directory into VLC's extensions folder, restart. Activation: View $\to$ Extensions $\to$ Saturn Chat. No environment variables, no Python on the PATH, no API key.

On activation, the extension reads metadata from whatever is playing---title, artist, album, genre, timestamp---via VLC's Lua metadata API. A dialog opens with two dropdowns: discovered services and available models. The user types a question about the media and gets a response. Discovery and model enumeration run in the background; the user never sees them.

\subsubsection{Technical Architecture}

VLC's extension system supports only Lua scripting with restricted HTTP capabilities (GET requests only, no POST), and Lua has no native mDNS support. Two layers work around these constraints.

Layer 1: A Python/FastAPI bridge service (`vlc\_discovery\_bridge.py`) runs as a background subprocess launched by the Lua extension. The bridge performs mDNS discovery using the same `dns-sd` subprocess calls as the core Saturn client, browses `\_saturn.\_tcp.local.`, resolves service addresses, maintains health checks, and exposes an OpenAI-compatible REST API on localhost (127.0.0.1). When the Lua extension activates, it launches the bridge with automatic port assignment. The bridge writes its port to a file that the extension reads, establishing inter-process communication without hardcoding ports.

Layer 2: The Lua extension (`saturn\_chat.lua`) creates a VLC dialog window, manages user interaction, formats requests as URL-encoded JSON payloads (working around Lua's GET-only limitation), and sends them to the localhost bridge via HTTP. When the user deactivates the extension, the bridge is terminated.

The bridge is packaged into a standalone executable using PyInstaller, eliminating the Python dependency for end users. Cross-platform backgrounding handles Windows (`start /B`), macOS, and Linux process launching. The entire installation is a directory copy; deployment requires no system-level configuration.

\subsubsection{Significance}

VLC's audience watches video---not debugs APIs. If Saturn discovery works inside a media player, it works at the network infrastructure layer, not just in developer tooling. Under the hood the bridge calls \texttt{dns-sd -B \_saturn.\_tcp local} followed by \texttt{dns-sd -L}, the same sequence every other Saturn client runs. More broadly, the subprocess pattern shows how to retrofit discovery into software that Saturn's developers do not control: any application with a plugin or extension system can launch an identical bridge.

\section{GL.iNet Router Deployment}

\subsection{Hardware Constraints}

580~MB of RAM, 8~GB of flash, a single-core MIPS processor. The GL.iNet MT300N-V2 cannot run inference---but a Saturn beacon registers mDNS records and rotates API keys, work measured in kilobytes and milliseconds. This router already assigns IP addresses via DHCP and resolves names via DNS. Adding Saturn is one more line in its process table.

\subsection{OpenWRT Integration}

A custom OpenWRT package installs the beacon as a system service. Configuration goes through UCI (Unified Configuration Interface), the same tool an administrator uses for DHCP pools and firewall rules: enter the cloud credentials or local service URLs, and the beacon validates them at startup. Only Python's standard library is required, which keeps the footprint within the MT300N-V2's 8~GB flash.

\subsection{Significance}

What matters is not the router but the role it already fills. The box assigning IP addresses and resolving DNS names now also advertises AI endpoints. A parent on a home network or an IT person managing a small office adds Saturn to hardware they already maintain; every device that joins the WiFi discovers AI services alongside its IP address.

\subsection{Limitations}

Multi-site deployments break this model: two homes means two separate configurations; a campus would need multiple routers or centralized infrastructure Saturn does not yet provide. If the router itself fails, discovery stops until it reboots. These are edge-deployment constraints, not Saturn-specific failures.

\section{Evaluation Approach}

The first question is binary: can mDNS/DNS-SD provision AI without user configuration? Throughput percentiles and scalability curves measure how well; they do not answer whether. Three requirements pin down what ``works'' means.

\begin{description}
\item[R1: Service Discovery.] Saturn services can be found on the local network without endpoint URLs, API keys, or configuration files on the client side.
\item[R2: Connection Sufficiency.] The information carried in mDNS/DNS-SD records provides everything a client needs to establish a working API connection, with no out-of-band configuration.
\item[R3: Cross-Implementation Interoperability.] Independent implementations in different languages consume the same protocol records without a bridging or adaptation layer.
\end{description}

R1 asks the simplest version: can a client with no prior knowledge find and connect? R2 drills deeper---every connection concern (address, dialect, credentials, routing) must map to a protocol field or the system still depends on side channels. R3 checks whether independent codebases in different languages consume the same records without shared code.

\subsection{R2: Connection Sufficiency}
\label{sec:r2-sufficiency}

Knowing a service exists is not the same as being able to use it. A client still needs a network address, an API dialect, the right endpoint path, and a credential. If any of those travel through a README, a \texttt{.env} file, or a Slack message from IT, the system is not zero-configuration. Siddiqui et al.~\cite{siddiqui2012zeroconf} defined the standard as ``little end-user intervention''; connection sufficiency tests whether that standard survives past discovery into actual use.

The resolution sequence follows RFC~6763~\cite{rfc6763}, extended with AI-specific metadata in the TXT record:

\begin{enumerate}
\item \textbf{Browse (PTR).} The client multicasts a PTR query for \texttt{\_saturn.\_tcp.local.}. Each response names a Saturn service instance. Guttman~\cite{guttman2001zeroconf} described this mechanism as enabling ``direct communications between two or more computing devices via IP'' without configuration.
\item \textbf{Locate (SRV).} The client queries each instance's SRV record to get its hostname and port.
\item \textbf{Describe (TXT).} The TXT record carries key-value pairs: \texttt{api\_base} (endpoint URL for cloud deployments), \texttt{api\_type} (API compatibility identifier), \texttt{ephemeral\_key} (authentication credential), \texttt{auth} (authentication method), \texttt{priority} (routing preference), and \texttt{deployment} (cloud or network).
\item \textbf{Resolve (A/AAAA).} The client resolves the SRV target hostname to an IP address.
\end{enumerate}

Table~\ref{tab:txt-connection} maps each concern to its DNS record. No row requires out-of-band information.

Syed et al.~\cite{syed2025aiaas} documented the per-application credential burden of AI-as-a-Service; Saturn's TXT schema replaces it with a single network broadcast.

\paragraph{Limitation.} The TXT record schema describes how to \emph{connect} to a service but not what the service \emph{can do} at fine granularity. The optional \texttt{capabilities} field advertises broad categories (chat, code, vision), but per-model attributes---context window size, tool-use support, embedding dimensions---fall outside the discovery record. Clients that need this information query the \texttt{/v1/models} endpoint after connecting. This reflects a protocol constraint: mDNS TXT records are limited to 255~bytes per key-value pair~\cite{rfc6762}, and per-model metadata would exceed that budget.

\subsection{Methodology}

A cognitive walkthrough~\cite{wharton1994walkthrough} traces a user through an interface step by step, checking whether the correct action is identifiable at each point. We adapted it to compare Saturn against traditional API-key provisioning, starting from infrastructure setup and ending at a working AI feature in an end user's hands.

The GL.iNet router deployment (Chapter~4) demonstrates Saturn on real hardware; systematic latency testing on physical networks remains future work.

\section{Threat Models}
\label{sec:threat-models}

Saturn trades enterprise-grade verification for zero-configuration access. This is a deliberate choice for the target environment---campus WiFi, home networks, small offices---not an oversight. Four threat scenarios illustrate the trade-off space.

\subsection{Corporate Data Exfiltration}

An employee on a corporate network discovers a Saturn beacon advertising a cloud AI endpoint and sends proprietary data to the service. Saturn's beacon architecture limits this threat: beacons announce endpoints but do not proxy traffic---API requests flow directly from client to backend over HTTPS. An administrator who configures a cloud beacon accepts that users will send requests to that cloud provider, identical to the risk of any cloud-accessible API.

The distinction between \texttt{deployment=local} and \texttt{deployment=cloud} in Saturn's TXT records makes this risk visible. A security-conscious administrator can restrict Saturn to local deployments only, ensuring all AI traffic remains on the LAN. Client applications can filter by deployment type, refusing cloud endpoints when policy requires it.

\subsection{Untrusted Local Administrator}

The administrator who runs Saturn controls what services are advertised, what priority values are assigned, and (for cloud backends) what API keys are distributed. A malicious administrator could advertise a rogue endpoint, capture prompts, or redirect traffic.

This threat is real but not novel. A malicious network administrator already controls DNS, DHCP, firewall rules, and routing tables. Saturn does not introduce new capabilities for compromise---it inherits the standard LAN trust model. The critical design decision is that beacons do not proxy traffic: even if the administrator controls what services are discovered, actual API communication uses HTTPS between client and endpoint. The administrator cannot read encrypted traffic without also compromising the endpoint's TLS certificate.

Logging is the residual risk. An administrator running a local Ollama instance can access Ollama's request logs, which may contain prompts. This is a property of the backend, not of Saturn. Users concerned about prompt privacy should verify the backend's logging configuration independently.

\subsection{Rogue Service Registration}

Any device on a Saturn-enabled network can register mDNS services and advertise itself as an AI provider. An attacker on a coffee shop WiFi network could broadcast a \texttt{\_saturn.\_tcp.local.} service pointing to a malicious endpoint that captures prompts or returns manipulated responses.

mDNS does not authenticate service registrars. Kaiser and Waldvogel~\cite{kaiser2014privacy} documented this as a known limitation; it is not unique to Saturn. Saturn mitigates it through priority-based selection: administrator-configured beacons advertise low priority values (e.g., 10); rogue services default to higher values and rank lower in client selection. This raises the bar above trivial spoofing but does not prevent a determined attacker on the same segment.

For environments where rogue registration is a serious concern (public WiFi, shared networks), Saturn is not the appropriate discovery mechanism. The design explicitly targets administratively controlled networks where the set of legitimate service providers is managed.

\subsection{ISP-Level Service Injection}

An ISP pushes a firmware update to a customer's router that includes a Saturn beacon advertising the ISP's AI service with priority~0 (highest). Users on the network are silently directed to the ISP's endpoint, which may log prompts, inject advertising, or charge for access.

This scenario, explored through the design fiction in Chapter~5, is technically feasible. Saturn's priority mechanism would route users to the ISP-injected service ahead of locally configured alternatives. The attack requires firmware-level access---a capability ISPs already exercise for DNS redirection and captive portals.

Mitigation requires client-side policy. Applications could maintain a whitelist of trusted service providers, require user confirmation before connecting to new services, or display service provenance (local vs.\ cloud, priority source) before routing requests. Saturn's protocol exposes the metadata needed for these checks; implementing them is a client responsibility.
\section{Trust, Privacy, and Verification}
\label{sec:trust-privacy}

Saturn makes an explicit trust assumption: the local network is controlled by the administrator who runs Saturn. This assumption holds for home networks, small offices, and institutional networks where WiFi setup is managed centrally. It breaks down in hostile environments (open WiFi, shared accommodations, untrusted networks).

\subsection{Privacy of Broadcast Metadata}

The packet capture revealed 11 fields exposed in mDNS broadcasts: API base URL, API type, deployment context, priority, version, capabilities, context window, cost tier, features, models, and service type. This is more metadata than typical static configuration, where credentials live in \texttt{.env} files visible only to the local filesystem.

Three observations bound this exposure. First, service metadata is already inferrable on trusted networks---any user on the LAN can attempt connections to common ports and discover running services. Saturn makes this explicit through mDNS rather than leaving it to port scanning, but the information is not fundamentally new. Second, prompts and responses remain encrypted. Broadcast metadata reveals what services exist; HTTPS protects what users ask and what they receive. Third, for cloud deployments using ephemeral keys, the broadcast credential expires automatically within 10 minutes. An observer who captures the broadcast cannot use that credential days later.

For users in untrusted networks (shared WiFi, public venues), Saturn is inappropriate. For users in private networks (home, small office, institutional LANs with controlled access), the privacy trade-off is reasonable.

\subsection{Administrator Trust}

Saturn assumes the network administrator can be trusted. In institutional settings, this is typically the case. Administrators deploy and configure Saturn, control priority values, and manage rotations. A malicious administrator could configure rogue services, manipulate priorities, or redirect traffic---but a malicious administrator can do this through many channels (DNS hijacking, ARP spoofing, network monitoring). Saturn does not introduce new capabilities for compromise; it inherits the standard LAN trust model.

\section{Limitations and Generalizability}
\label{sec:limitations}

The evaluation establishes Saturn's feasibility under tested conditions. These conditions are specific; generalization requires care.

Multicast does not route across subnets. Saturn works on any single LAN segment (home WiFi, office LAN, dorm network, campus subnet). It does not provide discovery across multiple subnets without infrastructure (relay servers, tunneling). This is by design---zero-configuration at the edge, not across wide-area networks.

Step counts reflect what an LLM-based agent must do to complete tasks. Humans may take different paths, require different numbers of decisions, or find configuration easier or harder. The step reduction (65\%) is substantial enough that human paths would likely show similar reduction, but human-subject evaluation would strengthen claims about usability.

Each condition used $N=5$ trials for discovery and $N=1$ per condition for configuration. Confidence intervals are tight (5.7--12\% coefficient of variation), suggesting stability, but larger samples ($N=10$ or $N=20$) would strengthen statistical claims for publication.

Docker sandbox differs from real WiFi networks. Discovery latency measured in emulation may not reflect real-world behavior with physical networks, interference, and multi-hop scenarios. Real-network evaluation is needed before claiming production readiness.

The evaluation measured information leakage but did not simulate adversarial attacks. Active penetration testing (DNS rebinding, ARP spoofing, rogue service registration) would reveal practical exploitability beyond passive observation.

Despite these constraints, the evaluation demonstrates that zero-configuration discovery is feasible, that it reduces configuration work, and that information leakage is bounded and documentable. Saturn achieves its core design goal within its target scope (single-segment, administratively controlled networks).

\section{Relation to Prior Work}
\label{sec:prior-work}

Saturn positions itself at the intersection of three research areas: service discovery, AI provisioning, and network security.

\subsection{Service Discovery}

Zero-configuration service discovery via mDNS and DNS-SD is established in networking research. Guttman~\cite{guttman2001zeroconf} and Siddiqui~\cite{siddiqui2012zeroconf} documented the protocol foundations. Saturn applies these existing protocols to a new domain (AI services) without modifying them. This is intentional: reusing proven infrastructure avoids introducing new complexity or vulnerabilities.

The evaluation confirms that mDNS discovery is feasible for AI services. This extends prior work by demonstrating practical applicability beyond media servers and printers.

\subsection{AI-as-a-Service Provisioning}

Syed et al.~\cite{syed2025aiaas} documented the current architecture: users must obtain and manage credentials from each provider independently. Costa~\cite{costa2024democratization} and Gabriel~\cite{gabriel2024educational} discuss how this fragmentation barriers access for underfunded users and institutions. Saturn addresses this by making local services discoverable, reducing the number of external credentials needed.

Saturn does not replace cloud AI services. Instead, it complements them by enabling local deployment while maintaining compatibility with cloud APIs. Users can discover both local and cloud services through the same mechanism.

\subsection{Security Trade-offs}

Ward \& Beyer~\cite{ward2014beyondcorp} and Osborn~\cite{osborn2016beyondcorp2} advocate zero-trust architecture: assume the network is hostile, verify every request. Saturn deliberately diverges. Saturn assumes the network is controlled by the administrator and trusts mDNS broadcast. This is appropriate for edge networks (home, small office) where administrative control is clear. BeyondCorp is appropriate for large enterprises where network trust is weaker.

Kaiser \& Waldvogel~\cite{kaiser2014privacy} documented privacy concerns in mDNS (service enumeration, device naming). Saturn accepts this known exposure as the price of zero-configuration discovery. Their proposed privacy-preserving extensions could be layered on top if needed~\cite{kaiser2014efficient}.

\section{Future Work}
\label{sec:future-work}

The evaluation identifies several directions for future development:

Current Saturn uses multicast, limited to single segments. A relay service could forward mDNS queries across subnets, enabling discovery in multi-segment campus networks. This requires careful design to avoid broadcast storms and privacy leakage across organizational boundaries.

Kaiser \& Waldvogel's privacy-preserving mDNS-SD protocols could reduce metadata visibility. Rather than broadcasting all service metadata, beacons could advertise only service existence, with detailed metadata available through secure queries. This would prevent passive enumeration while maintaining zero-configuration setup for users with valid credentials.

Current Saturn requires an administrator to configure and run the central coordinator. A fully decentralized design could allow peer-to-peer deployment, where any user could provision local services. This trades operational simplicity for deployment flexibility.

Evaluate Saturn on real WiFi networks with interference, multi-hop scenarios, and physical distance. Docker sandbox results do not account for real radio propagation characteristics. Production deployment requires validation in realistic environments.

Test actual users (developers, students, researchers) configuring AI services with and without Saturn. Measure not just step count but cognitive load, time-to-first-success, and error recovery. This would validate that agent-measured step reduction translates to actual human usability improvement.

Conduct formal penetration testing: DNS rebinding attacks, ARP spoofing, rogue service registration, key exfiltration. Measure whether documented mitigations actually prevent exploitation. This would move from ``passive observation'' to active security validation.

\section{Broader Implications}
\label{sec:implications}

Saturn's core contribution is making AI services discoverable and automatically configurable within administrative domains. This addresses a real problem: users who lack technical expertise, funding, or network administrative privileges currently cannot access local AI services.

The evaluation demonstrates feasibility. Configuration effort drops 59\%; discovery succeeds 100\% of the time. Security exposure is bounded and documentable. These results suggest that zero-configuration AI provisioning is practical and worth deploying.

The broader implication is equity. Bassignana~\cite{bassignana2025aigap} documents the AI gap---the digital divide where those without resources cannot access cutting-edge AI tools. Gabriel~\cite{gabriel2024educational} shows how this affects educational opportunity. Carmona-Galindo~\cite{carmona2025hsi} specifically studies Hispanic-Serving Institutions, where budgetary constraints limit AI tool availability.

Saturn does not solve the global AI access problem. But within a networked institution, it enables any administrator to deploy AI services that all users can discover and use. It lowers barriers from ``hire a specialist to configure each tool'' to ``run a single daemon.'' This matters most for resource-constrained institutions---community colleges, public schools, libraries---where every dollar of IT budget counts.

\chapter{Saturn Protocol Specification}
\label{app:protocol-spec}

This appendix specifies the Saturn protocol as a standalone reference. An implementor reading only this appendix and the RFCs it cites should produce a system interoperable with any conforming Saturn deployment.

\section{Service Registration}

A Saturn beacon registers one DNS-SD service instance under the type \texttt{\_saturn.\_tcp.local.} (RFC~6763~\cite{rfc6763}). Registration creates three DNS records:

\begin{enumerate}
\item A \textbf{PTR record} mapping \texttt{\_saturn.\_tcp.local.} to the instance name (e.g., \texttt{ollama-lab.\_saturn.\_tcp.local.}).
\item An \textbf{SRV record} providing the beacon's hostname and port.
\item A \textbf{TXT record} carrying service metadata as key-value pairs.
\end{enumerate}

The instance name is a human-readable UTF-8 string chosen by the administrator. It must be unique within the multicast domain. The \texttt{.local.} suffix confines all queries and responses to the link-local scope defined by mDNS (RFC~6762~\cite{rfc6762}).

Every URL advertised in \texttt{api\_base} must implement three HTTP paths:

\begin{description}
\item[\texttt{GET /v1/health}] Returns JSON with a \texttt{status} field. Clients call this to verify liveness before use and periodically during sessions. A non-200 response or timeout (recommended: 5~seconds) indicates the service is unavailable.
\item[\texttt{GET /v1/models}] Returns a JSON object with a \texttt{data} array of model objects, each containing at minimum an \texttt{id} field. Follows the OpenAI list-models response format.
\item[\texttt{POST /v1/chat/completions}] Accepts a JSON body with \texttt{model} (string) and \texttt{messages} (array of \texttt{\{role, content\}} objects). Returns a chat completion response. Must support both non-streaming (JSON response) and streaming (server-sent events with \texttt{data:} prefix) modes, selected by the \texttt{stream} boolean in the request body.
\end{description}

For cloud beacons, \texttt{api\_base} points to the upstream provider, not the beacon. Clients authenticate with the provider using the \texttt{ephemeral\_key} from the TXT record.

\section{Discovery Algorithm}

Listing~\ref{lst:discovery} specifies client discovery in pseudocode.

\begin{verbatim}
function discover():
    instances = mdns_browse("_saturn._tcp.local.")
    services = []
    for instance in instances:
        srv = mdns_resolve_srv(instance)
        txt = mdns_resolve_txt(instance)
        if txt["version"].major != 1:
            skip  // incompatible protocol version
        services.append({
            host: srv.hostname,
            port: srv.port,
            metadata: txt
        })
    sort services by metadata["priority"] ascending
    for service in services:
        url = service.metadata["api_base"]
        headers = {}
        if "ephemeral_auth" in service.metadata["features"]:
            headers["Authorization"] =
                "Bearer " + service.metadata["ephemeral_key"]
        response = http_get(url + "/v1/health",
                            headers, timeout=5s)
        if response.status == 200:
            return service  // connected
    return null  // no healthy service found
\end{verbatim}
\label{lst:discovery}

Clients may cache discovery results. A recommended cache TTL is 60~seconds; clients should re-browse when the cached service fails a health check.

\section{Ephemeral Key Rotation Protocol}

Cloud beacons rotate API credentials on a fixed schedule to bound exposure windows. The rotation proceeds as follows:

\begin{enumerate}
\item At startup, the beacon creates a new API key via the provider's provisioning API and records its creation time $t_0$.
\item The beacon publishes the key in the \texttt{ephemeral\_key} TXT field and sets \texttt{rotation\_interval} to the rotation period $R$ (default: 300~seconds).
\item At $t_0 + R$, the beacon creates a new key $k_1$, updates the TXT record to advertise $k_1$, and starts a grace timer for the old key $k_0$.
\item After a grace period $G$ (default: 5~seconds), the beacon deletes $k_0$ from the provider's system.
\item Steps 3--4 repeat every $R$ seconds.
\item On shutdown, the beacon deletes the current key from the provider.
\end{enumerate}

The key lifetime $L$ (default: 600~seconds) exceeds the rotation interval $R$, so at most two keys are valid simultaneously during the overlap window. This guarantees zero-downtime transitions: clients using the previous key continue working while the TXT record advertises its replacement.

\section{Conformance}

A conforming Saturn beacon must:
\begin{itemize}
\item Register exactly one \texttt{\_saturn.\_tcp.local.} service instance per advertised endpoint.
\item Include all six required TXT fields (Table~\ref{tab:appendix-txt}).
\item Include \texttt{ephemeral\_key} and \texttt{rotation\_interval} when \texttt{features} contains \texttt{ephemeral\_auth}.
\item Deregister the mDNS service on shutdown.
\item Delete active ephemeral keys from the provider on shutdown.
\end{itemize}

A conforming Saturn client must:
\begin{itemize}
\item Discover services via mDNS PTR query for \texttt{\_saturn.\_tcp.local.}.
\item Parse all required TXT fields; ignore unrecognized fields.
\item Sort discovered services by \texttt{priority} ascending.
\item Verify service health via \texttt{GET /v1/health} before use.
\item Fall back to the next-priority service when the current service becomes unreachable.
\end{itemize}

\section{Confidence Interval Computation}

All confidence intervals use Student's $t$ distribution. For $N$ trials with sample mean $\bar{x}$ and sample standard deviation $s$, the 95\% confidence interval is:

\[
\bar{x} \pm t_{0.025, N-1} \cdot \frac{s}{\sqrt{N}}
\]

\noindent where $t_{0.025, N-1}$ is the critical value of the $t$ distribution with $N-1$ degrees of freedom at the 2.5\% tail. For $N=5$ ($df=4$), $t_{0.025,4} = 2.776$.

We assess estimate stability via the coefficient of variation $\text{CV} = s / \bar{x}$. A CV below 10\% indicates low variance sufficient for thesis reporting. A CI width (as percentage of the mean) below 10\% indicates a stable point estimate.
